Writing Custom Middleware

Create a new middleware plugin by adding a single .mjs file to src/middlewares/. The loader discovers it automatically, validates its interface, and registers it in the database.

Step 1: Create the File

Create a new .mjs file in app/src/middlewares/. The filename becomes the file_name column in the database and is used for loading.

// app/src/middlewares/my-plugin.mjs

export default {
  // Required: unique middleware name
  name: 'my-plugin',

  // Optional: human-readable description
  description: 'Demonstrates the middleware interface',

  // Optional: semver version string (default: '1.0.0')
  version: '1.0.0',

  // Required: 'pre', 'post', or 'both'
  type: 'both',

  // Optional: if true, after() runs even for streaming responses
  supportsStreaming: false,

  // Optional: default settings (merged with per-tier/model overrides)
  defaultSettings: {
    enabled: true,
    myOption: 'value',
  },

  // Required for type 'pre' or 'both'
  async before(ctx, settings) {
    if (!settings.enabled) return;
    // Inspect or modify ctx.messages, ctx.params
    // Set ctx.abort = true to stop the pipeline
  },

  // Required for type 'post' or 'both'
  async after(ctx, settings) {
    if (!settings.enabled) return;
    // Inspect ctx.response, ctx.usage
    // Modify ctx.response to alter the output
  },
};

Step 2: Export the Required Interface

The middleware loader (middleware-loader.mjs) validates every imported module against these rules:

FieldTypeRequiredValidation
name string Yes Must be a non-empty string. Used as the unique identifier in the database.
type string No Must be 'pre', 'post', or 'both'. Defaults to 'both' if omitted.
before async function If type is pre or both Must be a function. Skipped modules log a warning.
after async function If type is post or both Must be a function. Skipped modules log a warning.
description string No Stored in DB for dashboard display.
version string No Defaults to '1.0.0'.
supportsStreaming boolean No Defaults to false. When false, the after() hook is skipped for streaming responses.
defaultSettings object No Defaults to {}. Stored as JSON in the DB; merged with per-tier/model overrides at runtime.
Validation failures are non-fatal. If a file fails validation (missing name, wrong type, missing required hook), it is skipped with a warning log. Other middlewares continue loading.

Step 3: The Context Object

Both before(ctx, settings) and after(ctx, settings) receive a context object built by middleware-runner.mjs. Here are all fields:

Mutable (pre-middlewares can modify)

FieldTypeDescription
ctx.messages Array The conversation messages array. Pre-middlewares can add, remove, or modify messages.
ctx.params Object Additional request parameters (temperature, max_tokens, etc.).

Read-Only Request Info

FieldTypeDescription
ctx.model string The resolved model name.
ctx.tier string The tier name (if the request was routed via tier).
ctx.apiKeyId number The authenticated API key's database ID.
ctx.agentName string Agent name from the X-Soul-Agent header.
ctx.sessionId string Session ID, derived from the key + agent + timeout window.
ctx.isStreaming boolean Whether the client requested streaming (stream: true).
ctx.authCtx Object Authentication context: { api_key_id, rpm_limit, tpm_limit, key_daily_budget, soul_id }.

State & Response

FieldTypeDescription
ctx.metadata Object Shared state between before() and after(). Use this to pass data between hooks.
ctx.response string|null The LLM response content. null in pre-hooks; populated in post-hooks.
ctx.isChunk boolean Whether this is a streaming chunk (for streaming-capable middlewares).
ctx.usage Object|null Token usage: { prompt_tokens, completion_tokens, total_tokens }. Only set in post-hooks.

Abort Control

FieldTypeDescription
ctx.abort boolean Set to true to stop the pipeline. No further middlewares run.
ctx.abortStatus number HTTP status code for the abort response. Default: 400.
ctx.abortMessage string Error message returned to the client (for error aborts).
ctx.abortResponse Object|null When set with abortStatus: 200, the pipeline sends this as a success response (used by the cache middleware to return cached results).

Step 4: Aborting Requests

There are two ways to abort the pipeline from a before() hook:

Error Abort

Block a request with an error status (e.g., 400, 429):

async before(ctx, settings) {
  if (someBadCondition) {
    ctx.abort = true;
    ctx.abortStatus = 429;
    ctx.abortMessage = 'Rate limit exceeded';
    ctx.metadata.errorType = 'rate_limit_exceeded';
    return;
  }
}

Success Abort (Cache Hit)

Return a pre-computed response without calling the LLM:

async before(ctx, settings) {
  const cached = lookupCache(ctx);
  if (cached) {
    ctx.abort = true;
    ctx.abortStatus = 200;
    ctx.abortResponse = {
      content: cached.content,
      stopReason: 'stop',
      usage: cached.usage,
      headers: { 'X-Cache': 'HIT' },
    };
    return;
  }
}
Error handling: If a middleware's before() or after() throws an uncaught exception, the runner logs the error and continues to the next middleware. Middleware errors are non-critical by design — they never crash the request pipeline.

Step 5: Settings Merging

The settings object passed to your hooks is the result of merging two layers:

  1. Default settings: The defaultSettings object from your middleware export, stored in the middlewares database table.
  2. Override settings: Per-tier or per-model overrides, stored in the middleware_assignments join table when a middleware is assigned via the API.

The merge is a shallow spread: { ...defaults, ...overrides }. Override keys replace default keys at the top level.

// Your middleware declares:
defaultSettings: { enabled: true, threshold: 100, mode: 'warn' }

// Admin assigns to a tier with override:
{ "threshold": 50 }

// Your hook receives:
settings = { enabled: true, threshold: 50, mode: 'warn' }

Step 6: Reload Without Restart

After adding or modifying a middleware file, you can reload the middleware registry without restarting the entire gateway:

POST /api/v1/middlewares/rescan

Re-scans the src/middlewares/ directory, imports all .mjs files (with cache-busting via a timestamp query parameter), validates and upserts them into the database. Also marks any previously-discovered middlewares whose files have been removed as undiscovered.

curl -X POST http://localhost:8042/api/v1/middlewares/rescan

Alternatively, restart the gateway:

ploinky restart soul-gateway

Step 7: Assign to Tiers or Models

A middleware is only active if it is assigned to at least one tier or model config. Use the management API to create assignments with optional setting overrides and a sort_order to control execution sequence.

Tier middlewares run first (broad policies), then model middlewares (specific overrides). Within each level, middlewares execute in ascending sort_order.

Execution order: Pre-middlewares run in order: tier middlewares (ascending sort_order), then model middlewares (ascending sort_order). Post-middlewares run in the same order. The cache middleware should always have the highest sort_order among pre-middlewares so it hashes the final message state.

Streaming Considerations

By default, after() hooks are skipped for streaming responses because chunks have already been sent to the client and modifications cannot reach them.

Set supportsStreaming: true if your after() hook performs observation-only work (logging, tracking, recording) that is still useful for streaming. The ctx.isStreaming flag lets you branch behavior:

async after(ctx, settings) {
  if (ctx.isStreaming) {
    // Observation only — ctx.response modifications won't reach the client
    recordMetrics(ctx);
    return;
  }
  // Non-streaming: can modify ctx.response
  ctx.response = sanitize(ctx.response);
}
