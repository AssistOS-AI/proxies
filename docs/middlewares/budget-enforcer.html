Budget Enforcer Middleware

Enforces daily spending budgets per API key, blocking requests before they incur cost and tracking spend in real time.

The budget enforcer works in two phases: a pre-check that blocks requests when the daily budget is exhausted, and a post-hook that updates the spend cache after each response so subsequent requests see accurate totals without querying the database.

Pre-Check: Budget Verification

The before() hook calls checkBudget(authCtx) from the cost throttler module (pipeline/cost-throttler.mjs). This function:

  1. Checks if the API key has a daily budget configured (key_daily_budget on the auth context). If no budget is set (null), the check passes immediately.
  2. Queries the daily spend for the key via getDailySpend().
  3. If the spend meets or exceeds the budget, throws a BudgetExceededError.

The middleware catches BudgetExceededError and converts it to a pipeline abort:

ctx.abort = true;
ctx.abortStatus = 429;
ctx.abortMessage = err.message;
ctx.metadata.errorType = 'budget_exceeded';

The client receives a 429 (Too Many Requests) response with the budget exceeded message.

Override Budget

The overrideDailyBudget setting allows overriding the key's configured budget at the tier or model level. When set, the middleware creates a modified auth context:

const authCtx = settings.overrideDailyBudget
  ? { ...ctx.authCtx, key_daily_budget: settings.overrideDailyBudget }
  : ctx.authCtx;

This allows tiers to have stricter (or more generous) budgets than the key's global setting.

In-Memory Spend Cache

The cost throttler maintains an in-memory spend cache to avoid querying the database on every request. The cache is keyed by "key:<api_key_id>".

Cache Lifecycle

  1. Cold start: On the first budget check for a key, the cache is empty. The throttler queries the call_logs table for all successful (status 200), non-free requests since the start of today (UTC midnight).
  2. Cached reads: Subsequent checks within the TTL window return the cached total without a database query.
  3. Post-response bump: After each response, the after() hook calls trackSpend(), which increments the cached total in memory.
  4. TTL expiry: After 10 seconds, the next check re-queries the database for accurate totals.
ParameterValueDescription
Cache TTL 10 seconds After this duration, the next budget check fetches fresh data from PostgreSQL.
Cache key format key:<id> One cache entry per API key ID.
Accuracy trade-off: The 10-second TTL means there is a brief window where a key could slightly exceed its budget if multiple requests arrive concurrently. The post-response bump minimizes this gap for sequential requests, but truly concurrent requests may both pass the check before either's cost is recorded.

Database Query

When the cache misses or expires, the throttler runs this SQL query:

SELECT COALESCE(SUM(total_cost), 0) as spent
FROM call_logs
WHERE api_key_id = $1
  AND started_at >= $2
  AND status_code = 200
  AND is_free IS NOT TRUE

Where $2 is the start of the current UTC day. This query:

  • Only counts successful requests (status 200).
  • Excludes free models (is_free IS NOT TRUE) so they never count against the budget.
  • Uses COALESCE to return 0 for keys with no spend history.

Post-Hook: Spend Tracking

The after() hook reads the total cost from ctx.metadata.totalCost (populated by the pipeline's cost calculation step) and calls trackSpend():

export function trackSpend(authCtx, totalCost, isFree) {
  if (isFree) return;
  bumpCache(authCtx.api_key_id, totalCost);
}

Key behaviors:

  • Free models excluded: If isFree is true, the spend is not tracked against the budget.
  • Cache bump only: The post-hook only updates the in-memory cache, not the database. The database is updated separately by the call logging step in the main pipeline.
  • Zero-cost skip: If totalCost is zero or falsy, the bump is skipped.

Settings

SettingTypeDefaultDescription
enabled boolean true Master switch. When false, both hooks return immediately.
overrideDailyBudget number|null null Override the key's daily budget for the assigned tier/model. When null, the key's own key_daily_budget is used. Set to a dollar amount to enforce a per-tier budget.

Integration with Cost Calculation

The budget enforcer depends on accurate cost data from the main pipeline. The cost calculation step (in the request handler) computes totalCost based on token usage and the model's pricing configuration, then stores it in ctx.metadata.totalCost along with the ctx.metadata.isFree flag. The budget enforcer's after() hook reads these values to update the spend cache.

If no cost data is available (e.g., the request failed before reaching cost calculation), the post-hook does nothing.
