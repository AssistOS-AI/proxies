Loop Detector Middleware

Detects agent loops by analyzing response similarity and conversation growth patterns, with configurable intervention strategies.

The loop detector solves a common problem with agentic LLM workflows: an agent gets stuck in a repetitive cycle, calling the same tools with the same arguments and receiving the same responses, burning tokens without making progress. The middleware detects this pattern and can intervene, block, or just log it.

Tracking ID Resolution

Each agent session is tracked by a unique ID. The resolveTrackingId() function uses the following priority:

  1. Session ID (ctx.sessionId) — preferred, since it represents a contiguous conversation from the same key + agent within the session timeout window.
  2. Fallback: apiKeyId:agentName — when no session ID is available, the key ID and agent name are concatenated with a colon. If both are missing, falls back to "unknown:unknown".

The tracking ID is stored in ctx.metadata._loopTrackingId so the after() hook can retrieve it without recomputing.

Response Fingerprinting

The core detection mechanism is response fingerprinting. Each LLM response is distilled into a short fingerprint that captures its action pattern rather than its full content:

  1. Extract tool names: A regex scans the response for JSON patterns like "name": "tool_name" (both OpenAI and Anthropic tool_use formats). All discovered tool names are sorted alphabetically.
  2. Extract content prefix: The first 200 characters of the response are trimmed and used as a content signal.
  3. Build raw fingerprint:
    • If tools were found: tools:toolA,toolB|first 200 chars
    • If no tools: just the first 200 characters
  4. Hash: The raw string is SHA-256 hashed and truncated to the first 16 hex characters.

This design means two responses that call the same tools in the same order with similar opening text produce identical fingerprints, even if the full content differs in details. This is intentional — loops often have minor variations in output but follow the same structural pattern.

Similarity Window and Threshold

The middleware maintains a sliding window of recent fingerprints per tracking session. Detection uses two signals:

Signal 1: Pure Response Similarity

If any single fingerprint appears similarityThreshold or more times within the last similarityWindow responses, a loop is detected with reason "response_similarity".

With defaults (similarityThreshold: 5, similarityWindow: 7), this means 5 out of the last 7 responses must have the same fingerprint.

Signal 2: Growth + Moderate Repetition

If the cumulative estimated token growth exceeds growthTokenThreshold and the most-repeated fingerprint accounts for more than 60% of the window, a loop is detected with reason "growth_and_repetition".

This catches slower loops where the agent is growing the context rapidly with repetitive patterns, even if the strict count threshold has not been hit.

Minimum responses: Loop detection requires at least 3 fingerprints in the session history. Fewer than 3 responses always returns { detected: false }.

Conversation Growth Tracking

The before() hook calls recordRequest() on every request, which:

  1. Increments the session's requestCount.
  2. Estimates the token count of the current messages array (using the heuristic of 4 characters per token) and adds it to the cumulative tokenGrowth.

Token growth is cumulative across the session, not per-request. This tracks how much total context the agent has consumed over its lifetime, which is one half of Signal 2.

Three Modes

When a loop is detected, the mode setting determines the response:

Mode: intervene (default)

Injects a system message at the beginning of the conversation to help the LLM break out of the loop:

ctx.messages = [
  { role: 'system', content: settings.interventionMessage },
  ...ctx.messages,
];

The default intervention message reads:

[Soul Gateway] Loop detected: your last several responses followed the same pattern. Stop the current approach and try something different.

After injecting, the middleware calls markIntervention() to increment the session's intervention count for tracking purposes. The request continues to the LLM with the extra guidance.

Mode: block

Aborts the request with HTTP 429 (Too Many Requests):

ctx.abort = true;
ctx.abortStatus = 429;
ctx.abortMessage = 'Agent loop detected (response_similarity): repeated response pattern over 7 requests';

This is the strictest mode. The agent receives an error and must decide how to proceed.

Mode: log

Does nothing beyond logging. The loop is detected and logged (by the checkLoop() function's internal log.warn()), but the request proceeds normally. Useful for observability without any impact on agent behavior.

Post-Dispatch: Recording Fingerprints

The after() hook records the response fingerprint for use in future loop detection. It runs after every successful response (including streaming), using the tracking ID stored in ctx.metadata._loopTrackingId during the before() phase.

The fingerprint array is capped to similarityWindow entries — older fingerprints are evicted from the front when the window is full.

Session Eviction

In-memory tracking sessions are evicted by a background timer:

ParameterValueDescription
Eviction interval 5 minutes How often the cleanup timer runs.
Max age 30 minutes Sessions not accessed within this window are deleted.

The timer is unref()'d so it does not prevent Node.js from exiting during graceful shutdown.
