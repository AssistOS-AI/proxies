Cache Middleware

Returns cached responses for identical prompts, avoiding redundant LLM calls and saving both latency and cost.

How Prompt Hashing Works

The cache middleware computes a SHA-256 hash of the conversation to use as a cache key. The hash input is:

sha256( JSON.stringify(ctx.messages) + '||' + ctx.model )

This means the cache is keyed on both the exact message content and the model name. Different models never share cache entries, even for identical prompts.

Why sort_order matters: The cache middleware should always be assigned with the highest sort_order among pre-middlewares (the built-in description recommends 999). This ensures it runs after all other pre-middlewares have mutated ctx.messages, so the hash reflects the final state. If cache ran first, it would miss mutations from system-prompt-injector, context-compressor, or other pre-middlewares.

Cache Lookup

The middleware calls findCachedResponse(promptHash, model) from logs-dao.mjs, which queries the call_logs table for a previous successful response (status 200) with the same prompt_hash and model values.

The call_logs table stores the prompt_hash column for every logged request, populated during the normal request pipeline. There is no separate cache table — the call log itself serves as the cache store.

Cache Hit Response

When a cached entry is found, the middleware aborts the pipeline with a success response (status 200) and returns the cached data directly to the client:

ctx.abort = true;
ctx.abortStatus = 200;
ctx.abortResponse = {
  content: cached.response_content,
  stopReason: cached.stop_reason || 'stop',
  usage: {
    prompt_tokens: cached.prompt_tokens || 0,
    completion_tokens: cached.completion_tokens || 0,
    total_tokens: cached.total_tokens || 0,
  },
  headers: { 'X-Cache': 'HIT' },
  cacheHit: true,
  promptHash,
};

Response Headers

Cache hits include an X-Cache: HIT header so clients can distinguish cached responses from live LLM calls. The pipeline handler forwards the headers object from abortResponse to the HTTP response.

Usage Data

The cached usage values (prompt_tokens, completion_tokens, total_tokens) are replayed from the original call log entry. This means cache hits report the same token counts as the original request, but no actual tokens are consumed.

Non-Streaming Only

The cache middleware skips streaming requests (ctx.isStreaming === true) for two reasons:

  1. Format mismatch: Cached responses are stored as complete response objects, not as SSE chunk streams. Replaying them as a stream would require synthesizing SSE frames.
  2. Expectation mismatch: Streaming clients expect incremental delivery. Returning a cached blob in a single frame defeats the purpose of streaming.

Additionally, supportsStreaming is set to false, confirming the middleware has no after() hook and does not participate in post-dispatch processing.

TTL and Eviction

Cache entries live as long as the underlying call_logs rows exist. The log retention period is controlled by config.retentionDays (default: 90 days). There is no separate cache TTL — when log entries are cleaned up, their cached data goes with them.

To invalidate cache entries without waiting for retention, delete the relevant call_logs rows or change the request content (even a single character difference produces a completely different SHA-256 hash).
