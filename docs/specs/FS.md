# Functional Specification (FS)

## Document Information

| Field | Value |
|-------|-------|
| System | Soul Gateway |
| Version | 1.0 |
| Date | 2026-03-31 |

## Purpose

This document specifies the functional behavior of Soul Gateway. Each function maps to one or more user requirements (URS) and is realized by one or more design specifications (DS).

## Functions

| ID | Function | Description | Related URS |
|----|----------|-------------|-------------|
| FS01 | Request Dispatch | Receive an HTTP POST to `/v1/chat/completions` (or `/chat/completions`), extract the Bearer token from the Authorization header, authenticate the key against `api_keys` table via key hash lookup, parse the JSON body to extract `model`, `messages`, `stream`, and additional LLM parameters (tools, tool_choice, temperature, etc.), and route the request through the pipeline to an upstream provider. Return the response in OpenAI chat completion format. | URS01, URS02 |
| FS02 | Model Resolution | Given a requested model name, resolve it to a concrete provider and provider-model pair. First attempt direct lookup in `model_configs` by name. If no direct match or the entry is of type `tier`, perform tier-based resolution by iterating through the tier's `model_refs` in priority order, skipping models in cooldown or disabled. Follow `fallback_model` chains with cycle detection (visited set). Return the resolved model configuration including provider key, provider model, pricing, and concurrency limits. | URS01, URS09 |
| FS03 | Tier Fallback | When a dispatched request fails with a cooldown-triggering error (rate limit or payment required), place the model in cooldown for `cooldownDurationMs` (default 1 hour). Re-resolve the tier to get the next available model. Repeat up to `maxModelRetries` (default 5) times. Track all attempted models to prevent re-trying the same model. On non-cooldown errors that still have an error classification (model-level failures), perform immediate cascade without cooldown. | URS09, URS11 |
| FS04 | OAuth Credential Management | Support two OAuth flows for upstream provider authentication: RFC 8628 device flow (polling-based, for GitHub Copilot and Google Gemini) and PKCE authorization code flow (browser redirect, for Kiro, Codex, and Anthropic). Maintain a registry of auth adapters, each exposing `startAuth()`, `pollForToken()` or `exchangeCode()`, `refreshToken()`, and `getHeaders()`. Store credentials on disk per provider per account. Run a background refresh loop (60s interval) that proactively refreshes tokens before expiry. Support multi-account pooling with quota-driven rotation: mark exhausted accounts and rotate to next available, reset quotas at midnight UTC. | URS06 |
| FS05 | Pre-Dispatch Middleware Pipeline | Before dispatching to the upstream provider, load enabled middlewares assigned to the resolved tier and/or model config. Build a middleware context object containing mutable `messages` and `params`, plus read-only request metadata (model, tier, apiKeyId, agentName, sessionId, isStreaming, authCtx). Execute each middleware's `before()` function in sort-order (tier middlewares first, then model middlewares). If any middleware sets `ctx.abort = true`, short-circuit: return the abort status and message to the client (or, if `abortStatus === 200` with an `abortResponse`, return it as a successful response for cache hits). Merge default settings with per-assignment override settings. | URS07 |
| FS06 | Post-Dispatch Middleware Pipeline | After receiving the upstream response, execute each middleware's `after()` function with the context populated with `response`, `usage`, and cost metadata. For streaming responses, skip middlewares that don't declare `supportsStreaming` (chunks are already sent). For non-streaming responses, allow middlewares to modify `ctx.response` which propagates back to the result. Errors in individual middlewares are logged but do not abort the response. | URS07 |
| FS07 | Response Streaming | For streaming requests (`stream: true`), write SSE headers (`Content-Type: text/event-stream`, `Cache-Control: no-cache`), then consume chunks from the upstream generator. Re-encode each `text_delta` chunk as an OpenAI SSE chunk (`data: {...}\n\n`). Pass through `tool_calls_delta` chunks. On `done` chunk, send the finish chunk with usage data followed by `data: [DONE]\n\n`. Record time-to-first-byte (TTFB) from the first `text_delta`. For mid-stream errors, send an error chunk and `[DONE]`. For non-streaming requests, buffer all chunks, then return a complete JSON response with `chat.completion` object including tool_calls if present. | URS02, URS12 |
| FS08 | Cost Calculation | Calculate per-request costs using two pricing models. Token-priced models: `input_cost = (prompt_tokens / 1,000,000) * input_price`, `output_cost = (completion_tokens / 1,000,000) * output_price`, `total_cost = input_cost + output_cost`. Request-priced models: `total_cost = request_cost` (flat per-request fee). Round all costs to 6 decimal places. Fall back to OpenRouter pricing lookup when a model has no pricing configured in the database. | URS04 |
| FS09 | Budget Enforcement | Check daily and monthly budgets for the API key before dispatch (via budget-tracker middleware). Query the sum of `total_cost` from `call_logs` for the current period. If the spent amount exceeds the key's budget, abort the request with HTTP 429 and `budget_exceeded` error type, including Retry-After header. Track post-dispatch costs by updating the running total. Skip budget tracking for requests resolved to free models (`is_free = true`). | URS04 |
| FS10 | Log Recording | After each request completes (success or failure), insert a row into the `call_logs` partitioned table. Record: timestamps (started_at, completed_at), API key ID, soul ID, agent name, session ID, requested and resolved model, streaming flag, full request messages (JSONB), response content, status code, stop reason, error details, latency, TTFB, token counts, costs, retry details, blacklist match info, truncation and slow flags, prompt hash, cache hit flag, and applied middlewares list. Use safe insert (catch and log DB errors without propagating to client). | URS05 |
| FS11 | Real-Time Log Broadcasting | After inserting a log entry, broadcast it to all connected WebSocket subscribers on `/ws/v1/logs`. Also broadcast to soul-specific subscribers filtered by `soul_id`. Support SSE as an alternative transport. Use 15-second ping interval on WebSocket connections to survive Cloudflare tunnel timeouts. | URS05 |
| FS12 | API Key Management | Provide CRUD API endpoints for API keys under `/api/v1/keys`. On creation, generate a random key prefixed with `sk-soul-`, store an SHA-256 hash for lookup and AES-256 encrypted key for recovery. Support configurable fields: label, rpm_limit, tpm_limit, daily_budget, monthly_budget, expires_at. Support revocation (is_revoked flag). Track last_used_at on each authentication. | URS06 |
| FS13 | Provider Management | Provide CRUD API endpoints for provider configurations under `/api/v1/providers`. Each provider has: name, display_name, protocol (openai/anthropic), base_url, encrypted API key, billing_type (api_key/free), auth_type (api_key/managed), and enabled flag. For managed auth providers, expose auth flow endpoints: start, poll, callback, status, remove account, reset quota. Auto-provision provider and model rows from adapter templates after first OAuth login. | URS01, URS06 |
| FS14 | Content Filtering | Before dispatch, scan all message content (concatenated from all messages, handling both string and array content parts) against enabled blacklist rules. Support three match types: `exact` (full content equality), `substring` (content includes pattern), and `regex` (case-insensitive regex test). On match, throw `BlacklistError` with HTTP 400, the rule ID, and a truncated (50-char) pattern excerpt for audit. | URS08 |
| FS15 | Agent Loop Detection | Implement behavioral loop detection as a middleware. In `before()`: track conversation growth (message count and estimated token count) per tracking ID (session ID or apiKeyId:agentName). In `after()`: extract a response fingerprint (sorted tool names + first 200 chars of content, hashed). If `similarityThreshold` (default 5) of the last `similarityWindow` (default 7) fingerprints are identical, flag as loop. Support three modes: `intervene` (prepend warning system message), `block` (abort with 429), `log` (record only). | URS08 |
