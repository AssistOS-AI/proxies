Middleware

A processing plugin that runs before and/or after LLM dispatch, auto-discovered from .mjs files and assignable to tiers or individual models.

What It Is

A middleware is a self-contained plugin that intercepts the request/response flow of the request pipeline. Middlewares can modify messages before they reach the LLM, inspect or transform the response after dispatch, enforce policies, cache results, or short-circuit the pipeline entirely.

Middleware Types

TypeRequired MethodsDescription
pre before() Runs only before LLM dispatch. Can modify messages, abort the request, or set metadata.
post after() Runs only after LLM dispatch. Can inspect the response, update metrics, or modify output.
both before() + after() Wraps the entire dispatch. before() runs pre-dispatch, after() runs post-dispatch.

Middleware Interface

Each middleware is a .mjs file that exports a default object with these properties:

PropertyTypeRequiredDescription
name string Yes Unique name for the middleware.
description string No Human-readable description.
version string No Semantic version (default: "1.0.0").
type string No "pre", "post", or "both" (default: "both").
supportsStreaming boolean No Whether the middleware correctly handles streaming responses.
defaultSettings object No Default configuration values that can be overridden per-assignment.
before(ctx) function If type is pre or both Pre-dispatch hook. Receives and can mutate the middleware context.
after(ctx, result) function If type is post or both Post-dispatch hook. Receives the context and the LLM response result.

Auto-Discovery

At startup, the middleware-loader.mjs module scans the src/middlewares/ directory:

  1. Lists all .mjs files in the directory.
  2. Dynamically imports each file and validates the exported interface (name must be a non-empty string, type must be valid, required methods must be functions).
  3. Caches the loaded module in memory for runtime use.
  4. Upserts a record in the middlewares database table (name, description, file_name, type, supports_streaming, default_settings, version).
  5. Marks any middlewares that exist in the database but no longer have a file on disk as "undiscovered".
// From middleware-loader.mjs — validation during scan
if (!mw || typeof mw.name !== 'string' || !mw.name) {
  log.warn(`Skipping ${file}: missing or invalid 'name'`);
  continue;
}
const type = mw.type || 'both';
if (!['pre', 'post', 'both'].includes(type)) {
  log.warn(`Skipping ${file}: invalid type '${type}'`);
  continue;
}
if (type !== 'post' && typeof mw.before !== 'function') {
  log.warn(`Skipping ${file}: type '${type}' requires before()`);
  continue;
}

Assignment

Middlewares are linked to tiers or individual models via the model_middlewares join table. Each assignment record contains:

ColumnDescription
model_config_id The tier or model this middleware is assigned to.
middleware_id Reference to the middleware in the middlewares table.
sort_order Execution priority within the assignment level (lower runs first).
settings JSON object of per-assignment settings that override the middleware's defaultSettings.
is_enabled Toggle to disable a specific assignment without removing it.

Execution Order

During a request, middlewares execute in this order:

  1. Tier-level middlewares (sorted by sort_order) — applied to the tier the model belongs to.
  2. Model-level middlewares (sorted by sort_order) — applied to the specific model config being used.

Pre-middlewares (before()) run in this order before LLM dispatch. Post-middlewares (after()) run in the same order after dispatch completes.

Abort Behavior

A pre-middleware can abort the pipeline by setting an abort status on the context. This can be either:

  • Success abort (status 200): The middleware provides a complete response (e.g., a cache hit), and the pipeline returns it directly without calling the LLM.
  • Error abort (status 4xx/5xx): The middleware blocks the request (e.g., rate limit exceeded, budget exhausted) and returns an error to the client.

Built-in Middlewares

Soul Gateway ships with 12 built-in middlewares:

FileNameTypeDescription
blacklist-scanner.mjs Blacklist Scanner pre Scans request content against blacklisted patterns.
budget-enforcer.mjs Budget Enforcer both Enforces daily and monthly budget limits per API key.
cache.mjs Cache both Prompt-hash based response caching. Returns cached responses as success aborts.
context-compressor.mjs Context Compressor pre Compresses large message contexts to fit within model limits.
loop-detector.mjs Loop Detector pre Detects repeated identical requests that may indicate an agent loop.
output-compressor.mjs Output Compressor post Compresses LLM response output.
rate-limiter.mjs Rate Limiter pre Per-key rate limiting with configurable RPM/TPM overrides.
request-logger.mjs Request Logger both Extended request/response logging.
response-filter.mjs Response Filter post Filters or sanitizes LLM response content.
session-context.mjs Session Context pre Injects session context from previous requests.
system-prompt-injector.mjs System Prompt Injector pre Prepends or appends system prompts to the messages array.
tpm-tracker.mjs TPM Tracker post Tracks token usage for TPM rate limiting.
