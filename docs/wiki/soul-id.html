Soul ID

A voluntary request header that identifies the human or system behind an API call, enabling per-soul log filtering and real-time WebSocket subscriptions.

What It Is

The Soul ID is a string value sent via the X-Soul-Id HTTP header on each request to /v1/chat/completions. It represents the identity of the end user, agent, or system making the call. Unlike the API Key, which authenticates the request, the Soul ID is an informational label — it is not verified or authenticated by the gateway.

Default value: If the X-Soul-Id header is not provided, the pipeline assigns the value "anonymous".

How It Flows Through the Pipeline

The Soul ID is extracted early in the request pipeline, during the authentication step:

  1. The authenticate() function in auth.mjs reads req.headers['x-soul-id'] and falls back to "anonymous".
  2. The value is returned as part of the auth context object (authCtx.soul_id).
  3. The pipeline stores it in logEntry.soul_id, which is persisted to the call_logs table in PostgreSQL.
  4. After the response completes, the log entry is broadcast to WebSocket subscribers via broadcastToSoul(authCtx.soul_id, broadcastEntry).
// From auth.mjs
const soulId = req.headers['x-soul-id'] || 'anonymous';

return {
  api_key_id: keyInfo.id,
  rpm_limit: keyInfo.rpm_limit,
  tpm_limit: keyInfo.tpm_limit,
  key_daily_budget: keyInfo.daily_budget != null ? Number(keyInfo.daily_budget) : null,
  soul_id: soulId,
};

Purpose

Use CaseDescription
Log filtering The dashboard and logs API can filter call logs by soul_id, allowing you to see all requests made by a specific user or system.
WebSocket subscriptions Clients can subscribe to /ws/v1/soul/:id to receive real-time log events for a specific Soul ID. The pipeline calls broadcastToSoul() after each completed request.
Audit trail The soul_id column in call_logs links every LLM call to the identity that initiated it.

Related Headers

Soul Gateway recognizes three identification headers, all voluntary and unauthenticated:

HeaderPipeline FieldPurpose
X-Soul-Id soul_id Identifies the human or system behind the request.
X-Soul-Agent agent_name Identifies the AI agent or coding assistant making the call. If not set, the pipeline falls back to parsing the User-Agent and x-coding-assistant headers using pattern matching (recognizes Claude Code, Codex CLI, Aider, Cursor, Copilot, Cline, Windsurf, and others).
X-Soul-Session session_id Groups related requests into a logical session for analysis.

Agent Detection Fallback

When X-Soul-Agent is not provided, the parseAgentName() function attempts to identify the agent from request headers:

  1. Checks the x-coding-assistant header first (e.g., "aider-0.72.1" yields "aider").
  2. Matches the User-Agent string against known patterns for 13 agents including Claude Code, Codex CLI, Gemini CLI, Cursor, Copilot, Cline, Roo Code, Windsurf, and Amp.
  3. Falls back to extracting the first product token from the User-Agent, filtering out generic runtime tokens like node, python, curl.
  4. Returns "unknown" if no match is found.

Example Request

curl https://soul.example.dev/v1/chat/completions \
  -H "Authorization: Bearer sk-soul-..." \
  -H "X-Soul-Id: user-42" \
  -H "X-Soul-Agent: my-agent" \
  -H "X-Soul-Session: session-abc" \
  -H "Content-Type: application/json" \
  -d '{"model": "fast", "messages": [{"role": "user", "content": "Hello"}]}'
