Rate Limiting

Dual-mechanism traffic control using per-key RPM (requests per minute) and TPM (tokens per minute) with a sliding-window counter in PostgreSQL.

What It Is

Soul Gateway enforces two independent rate limiting mechanisms on every API key:

  • RPM (Requests Per Minute): Limits how many requests a key can make within a 1-minute sliding window. Enforced before LLM dispatch — blocked requests never reach the upstream provider.
  • TPM (Tokens Per Minute): Tracks total token usage within a 1-minute sliding window. Tracked after dispatch — excess usage is logged as a warning but does not block requests.

Default Limits

MechanismDefaultConfigurable Via
RPM 60 requests/minute Per-key rpm_limit column in api_keys table
TPM 100,000 tokens/minute Per-key tpm_limit column in api_keys table

These defaults are set in config.mjs (defaultRpmLimit: 60, defaultTpmLimit: 100000) and applied when creating new keys without explicit limits.

Sliding Window Implementation

Both RPM and TPM use a 1-minute sliding window backed by the rate_limit_state PostgreSQL table. This approach survives process restarts (unlike in-memory counters) and supports multi-instance deployments.

ColumnDescription
key Composite key: rpm:key:{keyId} or tpm:key:{keyId}. Primary key with unique constraint.
window_start The beginning of the current window (now - 60 seconds).
counter Number of requests (RPM) or tokens (TPM) in the current window.
updated_at Timestamp of the last update.

The counter uses an atomic upsert with conditional reset:

// From rate-limiter.mjs
INSERT INTO rate_limit_state (key, window_start, counter, updated_at)
VALUES ($1, $2, 1, now())
ON CONFLICT (key) DO UPDATE SET
  counter = CASE
    WHEN rate_limit_state.window_start < $2  -- window expired
    THEN 1                                    -- reset to 1
    ELSE rate_limit_state.counter + 1         -- increment
  END,
  window_start = CASE
    WHEN rate_limit_state.window_start < $2
    THEN $2                                   -- new window
    ELSE rate_limit_state.window_start        -- keep current
  END,
  updated_at = now()
RETURNING counter, window_start

This single SQL statement handles both the first request (insert) and subsequent requests (update with window rotation), all in one atomic operation.

RPM Enforcement

RPM is checked in the checkRateLimit() function, which runs before LLM dispatch:

  1. Compute the window start as now - 60,000ms.
  2. Upsert the RPM counter for this key with the atomic query above.
  3. If the returned counter exceeds rpmLimit, throw a RateLimitError.
  4. The error response includes:
    • HTTP status 429 Too Many Requests
    • Retry-After header set to 60 seconds
    • Error message: "Rate limit exceeded: {counter}/{limit} RPM for this API key"

TPM Tracking

TPM is tracked in the trackTokenUsage() function, which runs after LLM dispatch completes:

  1. Compute the window start as now - 60,000ms.
  2. Upsert the TPM counter, adding the total tokens from the response.
  3. If the counter exceeds tpmLimit, return { exceeded: true, current, limit }.
  4. The result is logged but the response is not blocked — TPM is advisory only.
Why TPM is non-blocking: Token counts are only known after the LLM response is complete. Blocking at that point would waste the already-consumed upstream tokens. Instead, TPM overages are logged for monitoring and alerting.

Middleware Override

The built-in rate-limiter middleware can override the per-key rate limits. When assigned to a tier or model, it can:

  • Set custom RPM/TPM thresholds via per-assignment settings that differ from the key's defaults.
  • Abort the request with a 429 status and a Retry-After header if the limit is exceeded.

Similarly, the tpm-tracker middleware handles the post-dispatch TPM tracking as a dedicated post-middleware.

Budget Enforcement

Rate limiting is separate from budget enforcement. While rate limits control throughput (requests and tokens per minute), budgets control spend (dollars per day or month). Budget enforcement is handled by the budget-enforcer middleware, which checks the key's daily_budget against accumulated costs in the call_logs table.
