API Key

Bearer token authentication for client requests, stored as a SHA-256 hash with per-key rate limits, budgets, and expiry controls.

What It Is

An API Key is the sole authentication mechanism for the Soul Gateway /v1/chat/completions endpoint. Every request must include a valid key in the Authorization: Bearer <key> header. Keys are standalone entities — there is no family or group concept; each key carries its own rate limits and budget settings.

Key Properties

PropertyTypeDefaultDescription
label string — Human-readable name for the key (e.g., "production", "dev-team").
key_hint string — First 12 characters + last 4 characters of the plaintext key (e.g., sk-soul-36cc...c01b), used for identification in the dashboard.
rpm_limit integer 60 Maximum requests per minute for this key. Enforced pre-dispatch.
tpm_limit integer 100,000 Maximum tokens per minute for this key. Tracked post-dispatch (non-blocking).
daily_budget decimal $2.00 Maximum daily spend in USD. Enforced by the budget-enforcer middleware.
expires_at timestamp null Optional expiry date. Expired keys are rejected during authentication.
is_revoked boolean false When true, the key is permanently disabled.
last_used_at timestamp null Updated on every successful authentication.

Storage and Security

API keys are stored with two representations for different purposes:

ColumnAlgorithmPurpose
key_hash SHA-256 Fast lookup during authentication. The incoming key is hashed and compared against this column.
encrypted_key AES-256 Reversible encryption for key recovery by administrators. Requires the ENCRYPTION_KEY environment variable.
One-time plaintext: The full plaintext key is returned exactly once at creation time. It cannot be retrieved afterward — only the key_hint is available for identification.

Authentication Flow

Every request to /v1/chat/completions goes through this authentication sequence:

  1. Extract the raw key from the Authorization: Bearer <key> header.
  2. Compute the SHA-256 hash of the raw key.
  3. Look up the hash in the api_keys table, filtering out revoked keys.
  4. If the key has an expires_at date that has passed, reject it.
  5. Update last_used_at to the current timestamp.
  6. Return the auth context containing api_key_id, rpm_limit, tpm_limit, key_daily_budget, and the Soul ID.
// From auth.mjs
export async function authenticate(req) {
  const authHeader = req.headers['authorization'];
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    throw new AuthError('Missing or invalid Authorization header');
  }
  const rawKey = authHeader.slice(7).trim();
  if (!rawKey) throw new AuthError('Empty API key');

  const keyInfo = await resolveApiKey(rawKey);
  if (!keyInfo) {
    throw new AuthError('Invalid, expired, or revoked API key');
  }

  const soulId = req.headers['x-soul-id'] || 'anonymous';

  return {
    api_key_id: keyInfo.id,
    rpm_limit: keyInfo.rpm_limit,
    tpm_limit: keyInfo.tpm_limit,
    key_daily_budget: keyInfo.daily_budget != null ? Number(keyInfo.daily_budget) : null,
    soul_id: soulId,
  };
}

Key Creation

Keys are created via the management API. The creation process:

  1. Generates a cryptographically random key (or accepts a custom one).
  2. Computes the SHA-256 hash for lookup.
  3. Encrypts the plaintext with AES-256 for recovery.
  4. Stores the key_hint (first 12 + last 4 characters).
  5. Returns the full plaintext key in the response — this is the only time it is available.

Key Management

OperationDescription
Update Modify label, daily_budget, rpm_limit, or tpm_limit.
Revoke Set is_revoked = true. The key can no longer authenticate.
Reset budget Set budget_reset_at to now, clearing the daily spend counter.
List Returns all keys with their current daily spend (computed from call_logs).
