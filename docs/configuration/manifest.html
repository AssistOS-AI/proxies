Ploinky Manifest

The manifest.json file is the contract between Soul Gateway and the Ploinky orchestrator. It declares the container image, commands, environment, ports, dependencies, and installation hooks.

Full Manifest

{
  "container": "node:20-slim",
  "agent": "bash /code/startup.sh",
  "cli": "bash /code/cli.sh",
  "about": "Soul Gateway -- LLM proxy with Soul Family policies, ...",
  "enable": ["postgres"],
  "profiles": {
    "default": {
      "env": { ... },
      "ports": ["0.0.0.0:8042:8042"],
      "install": "bash /code/install.sh"
    }
  }
}

Top-Level Fields

FieldValueDescription
container "node:20-slim" The OCI container image Ploinky pulls to create the agent container. Uses the official Node.js 20 slim image based on Debian.
agent "bash /code/startup.sh" The command Ploinky runs inside the container when starting the agent. This is the entrypoint — it copies code, installs dependencies, loads secrets, and starts the Node.js process.
cli "bash /code/cli.sh" The command run when a user invokes ploinky cli soul-gateway for interactive CLI access to the agent.
about (string) Human-readable description shown in ploinky list and the dashboard. Describes the agent's purpose.
enable ["postgres"] Array of infrastructure services this agent depends on. Ploinky ensures PostgreSQL is running before starting Soul Gateway. Supported services include postgres and keycloak.

Profiles

Profiles allow different configurations for different environments. Soul Gateway defines a single default profile that Ploinky uses unless overridden.

profiles.default.env

A map of environment variable declarations. Each entry has the following structure:

"VARIABLE_NAME": {
  "required": false,
  "default": "value",
  "description": "Human-readable description"
}

When required is true, Ploinky will refuse to start the agent unless the variable is set (either in the manifest default, via ploinky var, or in .ploinky/.secrets). When false, the default value is used if the variable is not provided.

See the Environment Variables page for the complete list of all 19 declared variables.

profiles.default.ports

Array of port mapping strings in the format bind:host:container:

"ports": ["0.0.0.0:8042:8042"]
ComponentValueMeaning
Bind address 0.0.0.0 Listen on all interfaces. In production behind Cloudflare tunnels, this can be narrowed to 127.0.0.1.
Host port 8042 Port on the host machine.
Container port 8042 Port inside the container where the Node.js server listens.

profiles.default.install

"install": "bash /code/install.sh"

The install command runs once when Ploinky first creates the container (during ploinky start if no existing container is found). It:

  1. Creates persistent storage directories under /shared/soul-gateway/
  2. Copies the application code from /code/app/ to /app/
  3. Runs npm install --production to install Node.js dependencies
  4. Generates a random AES-256 encryption key if one does not already exist

How Ploinky Uses the Manifest

When you run ploinky start soul-gateway, the following sequence occurs:

  1. Dependency resolution: Ploinky reads the enable array and ensures all listed services (e.g., postgres) are running.
  2. Container creation: If no container exists, Ploinky pulls the container image (node:20-slim) and creates a new container with the declared env vars and port mappings.
  3. Variable injection: Ploinky merges manifest defaults with ploinky var overrides and .ploinky/.secrets, then injects them as container environment variables.
  4. Volume mounts: Ploinky mounts the repo directory at /code and the shared persistent storage at /shared.
  5. Install phase: On first creation, Ploinky runs the install command inside the container.
  6. Agent start: Ploinky executes the agent command (bash /code/startup.sh) as the container's main process.
Restart behavior: When running ploinky restart soul-gateway, Ploinky first calls syncCoreDependencies() to rsync the latest achillesAgentLib from Ploinky's own node_modules into the agent's workspace. This ensures dependency updates propagate without a full container rebuild.

Directory Layout Inside the Container

PathPurpose
/code/ Read-only mount of the Soul Gateway source repo. Contains app/, startup.sh, install.sh, etc.
/app/ Working copy of the application. Code is copied here during install and startup to allow npm to write node_modules.
/shared/soul-gateway/ Persistent storage that survives container recreation.
/shared/soul-gateway/config/encryption.key Auto-generated AES-256 encryption key (64 hex characters).
/shared/proxy_api_key Optional file containing the default proxy API key, loaded by startup.sh.
