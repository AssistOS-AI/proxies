Part 4

Middleware Framework

The pluggable middleware system: file-based discovery, the before/after hook interface, context object anatomy, execution ordering, abort mechanics, and the 12 built-in middlewares.

Overview

Soul Gateway's middleware framework allows modular processing logic to be inserted before and after the upstream LLM dispatch. Middlewares are discovered automatically from the middlewares/ directory on startup, registered in the database, and assigned to specific models or tiers via the model_middlewares join table.


  Request
    |
    v
  [Pre-middlewares: tier-level, then model-level]
    |  Can: mutate messages, abort (error or success), set metadata
    v
  Upstream Dispatch (retry + cooldown)
    |
    v
  [Post-middlewares: tier-level, then model-level]
    |  Can: observe response, modify content (non-streaming only), track metrics
    v
  Response to client

Plugin Discovery

The scanMiddlewares() function in middleware-loader.mjs runs during startup:

  1. Scans the src/middlewares/ directory for .mjs files
  2. Dynamic-imports each file with a cache-busting query parameter (?t=Date.now()) to force reimport on rescan
  3. Reads the default export and validates the middleware interface
  4. Caches the loaded module in memory (loaded Map keyed by file name)
  5. Upserts a row in the middlewares database table
  6. Marks any DB-registered middlewares whose files no longer exist on disk as is_discovered = false

Validation Rules

  • name must be a non-empty string
  • type must be 'pre', 'post', or 'both' (defaults to 'both')
  • If type is 'pre' or 'both', the module must export a before() function
  • If type is 'post' or 'both', the module must export an after() function

Middleware Interface

Each middleware module's default export must conform to this shape:

export default {
  // Required
  name: 'my-middleware',

  // Optional metadata
  description: 'What this middleware does',
  version: '1.0.0',
  type: 'both',               // 'pre' | 'post' | 'both'
  supportsStreaming: false,    // If true, after() runs even for streaming responses
  defaultSettings: {           // Default config, overridable per-assignment
    threshold: 100,
  },

  // Pre-dispatch hook (required if type is 'pre' or 'both')
  async before(ctx, settings) {
    // Mutate ctx.messages, set ctx.abort, populate ctx.metadata
  },

  // Post-dispatch hook (required if type is 'post' or 'both')
  async after(ctx, settings) {
    // Observe ctx.response, ctx.usage, modify ctx.response
  },
};

Execution Order

Middlewares are loaded and merged from two levels by loadMergedMiddlewares():

  1. Tier middlewares first: If the resolved model belongs to a tier, load middlewares assigned to the tier's model_config_id
  2. Model middlewares second: Load middlewares assigned to the specific model's model_config_id

Within each level, middlewares are sorted by sort_order (lower values run first, default 100). The final execution order is:

Tier middlewares (sorted by sort_order)
  then
Model middlewares (sorted by sort_order)

For pre-dispatch, only middlewares with type = 'pre' or type = 'both' are run. For post-dispatch, only those with type = 'post' or type = 'both'.

Streaming limitation: Post-dispatch middlewares with supports_streaming = false are skipped for streaming responses. Since SSE chunks have already been sent to the client, non-streaming middlewares cannot modify the response. Only streaming-capable middlewares run (for observation and logging purposes).

Abort Mechanism

A pre-middleware can abort the request in two ways by setting ctx.abort = true:

Error Abort

Set ctx.abortStatus to an error code (e.g., 429, 400) and ctx.abortMessage to an error string. The pipeline returns the error response, logs the abort, and terminates. Optional ctx.metadata.retryAfter sets the Retry-After header.

// Example: rate limiter abort
ctx.abort = true;
ctx.abortStatus = 429;
ctx.abortMessage = 'Rate limit exceeded';
ctx.metadata.retryAfter = 60;

Success Abort (Cache Hit)

Set ctx.abortStatus = 200 and ctx.abortResponse to an object with { content, stopReason, usage, cacheHit, promptHash, headers }. The pipeline treats this as a successful response: it sends the content to the client, calculates cost, inserts a log entry, and broadcasts -- all without dispatching to an upstream provider.

// Example: cache middleware success abort
ctx.abort = true;
ctx.abortStatus = 200;
ctx.abortResponse = {
  content: cachedContent,
  stopReason: 'stop',
  usage: { prompt_tokens: 0, completion_tokens: cachedTokens, total_tokens: cachedTokens },
  cacheHit: true,
  promptHash: hash,
};

Error Handling in Middlewares

If a middleware's before() or after() throws an exception, the error is logged but the pipeline continues to the next middleware. Middleware errors are non-critical by design -- they should never break a request.

Settings Merging

Each middleware has defaultSettings (defined in the module and stored in the middlewares table). When a middleware is assigned to a model/tier via model_middlewares, the assignment can provide override settings. At execution time, mergeSettings() performs a shallow merge:

const effective = { ...defaultSettings, ...overrideSettings };

Both values are parsed from JSON if stored as strings. This allows the same middleware to have different behavior per model -- for example, a rate limiter with different thresholds for different tiers.

Built-in Middlewares

Soul Gateway ships with 12 middleware plugins in the src/middlewares/ directory:

MiddlewareTypeDescription
rate-limiter pre Enforces per-key RPM and TPM limits using the rate_limit_state table. Aborts with 429 and Retry-After header.
budget-enforcer both Pre: checks daily/monthly budget against aggregated call_logs spend. Post: tracks the cost of the completed request.
blacklist-scanner pre Scans prompt messages against blacklist_rules (exact, substring, regex matching). Aborts with 400 on match.
cache both Pre: computes prompt hash and checks for a recent successful response with the same hash. Aborts with status 200 on cache hit. Post: no-op (the log insertion serves as cache storage).
loop-detector pre Detects repeated identical requests from the same API key within a short window. Aborts with 429 to prevent runaway agents.
context-compressor pre Compresses long message histories to fit within context windows by summarizing or truncating older messages.
system-prompt-injector pre Injects or modifies system prompts based on configurable templates.
session-context pre Loads recent session history and prepends it to the messages array for multi-turn context.
tpm-tracker post Tracks tokens per minute for rate limiting enforcement.
request-logger both Enhanced logging: logs request/response details for debugging.
response-filter post Scans response content and filters or flags problematic output.
output-compressor post Compresses response content for storage efficiency in logs.
