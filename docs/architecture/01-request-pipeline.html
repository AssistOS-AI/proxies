Soul Gateway Architecture
Part 1

Request Pipeline

The full lifecycle of a chat completion request, from HTTP ingress through authentication, routing, dispatch, streaming, cost calculation, and real-time broadcast.

Overview

Every request to /v1/chat/completions (or its alias /chat/completions) enters the pipeline() function in pipeline/pipeline.mjs. The pipeline is a single async function that orchestrates 13 discrete steps in sequence. Additional routes exist for the Anthropic Messages API (/v1/messages) and OpenAI Responses API (/v1/responses), each handled by dedicated proxy functions.

HTTP Routing

The HTTP server is created in server.mjs using Node's built-in http.createServer. Routes are matched by pathname and method in a flat if-else chain:

MethodPathHandler

CORS preflight is handled by handleCors() at the top of the request handler. WebSocket upgrade requests are skipped in the HTTP handler and caught by the server's 'upgrade' event, which delegates to handleUpgrade().

Step 1: Authentication

The authenticate() function in pipeline/auth.mjs extracts the Bearer token from the Authorization header. It calls resolveApiKey(rawKey) which hashes the key with SHA-256 and looks it up in the api_keys table by key_hash. Keys that are revoked or expired are rejected.

The returned context object contains:

  • api_key_id -- UUID of the matched key
  • rpm_limit -- requests per minute limit (default 60)
  • tpm_limit -- tokens per minute limit (default 100,000)
  • key_daily_budget -- daily spend limit in USD (default $2)
  • soul_id -- from the X-Soul-Id header, or "anonymous"

Step 2: Agent & Session Identification

The agent name is resolved from headers in priority order:

  1. X-Soul-Agent header (injected by achillesAgentLib's LLMClient.mjs from the AGENT_NAME env var)
  2. Parsed from User-Agent and X-Coding-Assistant headers (for IDE integrations)

The session ID is read from the optional X-Soul-Session header.

Step 3: Body Parsing & Validation

readJsonBody() reads the full request body and parses it as JSON. The pipeline validates that both model and messages fields are present. The stream flag is captured for later use. All additional parameters (e.g., tools, tool_choice, temperature) are passed through to the upstream provider via destructured llmParams.

Step 4: Model Routing

resolveModel(body.model) resolves the requested model name to a concrete provider and upstream model. See Part 6: Model Routing for the full algorithm. The returned modelInfo object includes the provider key, upstream model name, pricing data, concurrency limits, and optional tier context.

Steps 5-6: Pre-dispatch Middlewares & Prompt Hash

If the resolved model has a tier ID or model config ID, pre-dispatch middlewares are loaded and executed. See Part 4: Middleware Framework. Middlewares can:

  • Mutate the messages array (e.g., context compression, system prompt injection)
  • Abort with error (e.g., rate limiting, budget enforcement, blacklist blocking)
  • Abort with success (e.g., cache hit returning a cached response with status 200)

After middlewares run, the prompt hash is computed as SHA-256(JSON(messages) + "||" + resolvedModel). This uses the potentially-modified messages, so middleware mutations affect the hash.

Step 7: Dispatch with Retry & Cooldown

The dispatch loop has two levels of retry:

Level 1: Model-level retry (cooldown cascade)

Up to config.maxModelRetries (default 5) attempts. On each iteration:

  1. Acquire a concurrency slot via acquireModelSlot()
  2. Call dispatchWithRetry() for the current model
  3. On transient error (payment_required, rate_limit_error): put model in cooldown, re-resolve to get the next model in the tier
  4. On provider/model-level error: cascade immediately (no cooldown) to the next model
  5. Guard against resolving a previously-attempted model (tracks in attemptedModels Set)

Level 2: HTTP-level retry (exponential backoff)

dispatchWithRetry() in retry.mjs retries up to config.maxRetries (default 3) times with exponential backoff:

  • Base delay: 1000ms, multiplier: 2x, max: 30000ms, jitter: +/-20%
  • The generator is "peeked" -- the first chunk is consumed to verify connectivity. If the first .next() throws, the retry loop catches it.
  • Once chunks start flowing, retry is impossible -- mid-stream errors are passed through.
  • For managed OAuth providers, a 402 (payment required) triggers account rotation before retry.

Each retry attempt's status, error type, and delay are recorded in retriesDetail for the call log.

Steps 8-9: Stream Capture & Cost Calculation

If body.stream is true, tapStream() consumes the async generator and re-encodes each chunk as an OpenAI SSE event. Otherwise, handleNonStreaming() buffers all chunks and returns a single JSON response. Both functions return { content, usage, stopReason, ttfbMs, error }.

Cost is then calculated by calculateCost():

  • Token-based pricing: inputCost = (prompt_tokens / 1M) * inputPrice, same for output
  • Per-request pricing: flat requestCost regardless of tokens (used for Copilot and Kiro models)

Steps 10-13: Post-dispatch, Logging, Broadcast

Post-dispatch middlewares receive the response content, usage, and cost metadata. For streaming responses, only streaming-capable middlewares run (since chunks have already been sent to the client).

Response checks flag truncated responses (finish_reason !== 'stop') and slow requests (latency exceeding config.slowRequestMs, default 30s).

The complete log entry is inserted into the call_logs table (monthly-partitioned). Finally, broadcastLog() pushes a sanitized version to all WebSocket and SSE subscribers, and broadcastToSoul() sends the full entry to soul-specific subscribers.

Error Handling

All errors are caught by the outer try/catch in pipeline(). The error class hierarchy determines the HTTP response:

Error ClassHTTP StatusError TypeTrigger
AuthError401authentication_errorMissing/invalid/expired API key
RateLimitError429rate_limit_errorRPM or TPM exceeded
BudgetExceededError429budget_exceededDaily or monthly budget exceeded
BlacklistError400content_blockedPrompt matches blacklist rule
ModelNotFoundError404model_not_foundUnknown or disabled model
LoopDetectedError429loop_detectedExcessive repeated requests
QueueTimeoutError503queue_timeoutConcurrency slot wait exceeded 60s
UpstreamError502upstream_errorProvider returned error
SoulGatewayError500internal_errorBase class for all gateway errors

All SoulGatewayError subclasses carry a status, type, and optionally retryAfter. When retryAfter is set, the response includes a Retry-After header. Regardless of error type, the pipeline always attempts to insert a log entry (via safeInsertLog() which swallows DB errors) so failures are tracked.

Request ID

Every request is assigned a unique ID in the format chatcmpl-{uuid}. This ID is used in all SSE chunks and the final JSON response, matching the OpenAI convention for chat.completion objects.
