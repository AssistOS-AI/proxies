Model Routing

The resolution algorithm that maps a requested model name to a concrete provider endpoint: direct lookup, tier-based fallback chains, cooldown for transient errors, per-model concurrency queues, and pricing calculation.

Resolution Algorithm

The resolveModel(requestedModel) function in model-router.mjs follows a three-step resolution chain:


  resolveModel("axiologic-deep")
         |
    1. Direct model_config lookup by name
       - Found + type='model' + enabled? --> return buildModelInfo()
       - Found + type='tier'? --> fall through to step 2
       - Not found? --> fall through to step 2
         |
    2. Tier-based resolution (resolveFromTier)
       - Look up model_config with type='tier' and matching name
       - Iterate model_refs[] in priority order
         - Skip models in cooldown (isModelInCooldown)
         - Skip disabled models
         - Skip models missing provider_key/provider_model
       - First valid model? --> return buildModelInfo(model, tier)
       - No valid model? --> follow fallback_model chain
         |
    3. Fallback chain
       - tier.fallback_model points to another tier name
       - resolveFromTier(fallbackName, visited)
       - Cycle detection via visited Set
         |
    Not found? --> throw ModelNotFoundError (404)

Tier System

Tiers are stored in the model_configs table with type = 'tier'. They provide ordered model groups with automatic fallback.

Tier Fields

FieldPurpose
model_refsOrdered array of model names. The first enabled, non-cooldown model wins.
fallback_modelName of another tier to try if all models in model_refs are exhausted or in cooldown.
is_enabledSoft-disable for the entire tier.

Resolution Order within a Tier

Models in model_refs are tried strictly in array order (index 0 is highest priority). For each model name:

  1. Check if in cooldown (isModelInCooldown(modelName)) -- skip if yes
  2. Look up in model_configs by name
  3. Verify it is enabled and has provider_key + provider_model set
  4. First passing model is returned

Fallback Chains & Cycle Detection

When all models in a tier are exhausted (disabled, in cooldown, or misconfigured), the fallback_model field points to another tier name. The resolveFromTier() function follows this chain recursively. A visited Set prevents infinite loops:

async function resolveFromTier(tierName, visited = new Set()) {
  if (visited.has(tierName)) return null; // cycle detected
  visited.add(tierName);
  // ... try models, then follow fallback_model
}

Example chain: axiologic-deep --> models [A, B, C] --> all in cooldown --> fallback: axiologic-fast --> models [D, E] --> D available --> return D.

buildModelInfo()

Once a concrete model_config row is found, buildModelInfo() assembles the routing result:

{
  resolvedModel: string,       // The name passed back to the client
  modelConfigName: string,     // model_configs.name (may differ from resolvedModel for tiers)
  modelConfigId: UUID,         // model_configs.id
  providerKey: string,         // provider_configs.name
  providerModel: string,       // Model name sent to the upstream provider
  providerConfigId: UUID|null, // FK to provider_configs
  mode: string,                // 'deep', 'fast', etc.
  inputPrice: number,          // USD per 1M input tokens
  outputPrice: number,         // USD per 1M output tokens
  pricingType: 'token'|'request',
  requestCost: number,         // Flat cost for request-priced models
  isFree: boolean,             // Free models bypass budget limits
  maxConcurrency: number,      // Concurrency queue limit (default 3)
  tierName: string|null,       // Name of the tier if resolved via tier
  tierId: UUID|null,           // ID of the tier if resolved via tier
}

OpenRouter Pricing Fallback

If a model has zero input and output prices in the database and is token-priced, buildModelInfo() calls lookupOpenRouterPricing(providerModel) to fetch pricing from OpenRouter's API. This provides pricing data for models without explicit DB configuration.

Cooldown System

The cooldown system temporarily removes models from routing after transient errors (e.g., rate limits, payment required). It is implemented in model-cooldown.mjs using an in-memory Map.

Cooldown Entry

{
  expiresAt: number,     // Date.now() + cooldownDurationMs
  errorType: string,     // 'payment_required', 'rate_limit_error', etc.
  message: string,       // Original error message
  cooledAt: string,      // ISO timestamp when cooldown started
}

Triggers

shouldTriggerCooldown(errorClassification) returns true if the error type is in config.cooldownTriggers:

cooldownTriggers: ['payment_required', 'rate_limit_error']

Cascade Types

Two cascade behaviors exist in the pipeline's model-retry loop:

CascadeTriggerAction
Cooldown cascade shouldTriggerCooldown() returns true Model is put in cooldown for config.cooldownDurationMs (default 1 hour). The pipeline re-resolves to get the next model in the tier.
Immediate cascade shouldCascade() returns true (any error with classification) Model is skipped immediately without cooldown. Used for model/provider-level errors where the model might recover soon.

Cooldown Duration & Expiry

  • Default: 3,600,000 ms (1 hour), configurable via COOLDOWN_DURATION_MS env var
  • Auto-expiry: isModelInCooldown() checks the timestamp and deletes expired entries on read
  • Background cleanup: A setInterval every 5 minutes purges expired entries
  • Manual clear: clearCooldown(modelName) and clearAllCooldowns() for admin override

Dashboard API

getCooldownStatus() returns all active cooldowns with remaining time, used by the dashboard's cooldown panel:

Concurrency Queue

The model-queue.mjs module implements a per-model semaphore to limit concurrent requests to each upstream provider/model.

Semaphore Design

// Per-model state
{
  active: number,        // Currently running requests
  maxConcurrency: number,// Limit (from model_configs.max_concurrency, default 3)
  waiters: Array,        // Queue of pending requests
}

acquireModelSlot(resolvedModel, maxConcurrency)

  1. If active < maxConcurrency: increment active count, return release function immediately
  2. Otherwise: add to waiters queue with a 60-second timeout
  3. If timeout expires: reject with QueueTimeoutError (503, Retry-After: 10)
  4. When a slot becomes available, the next waiter is resolved

Release Behavior

The release function (returned by acquireModelSlot()) uses direct handoff: if there are waiters, the slot is passed directly to the next waiter without decrementing active. This prevents race conditions where a new request could steal the slot between decrement and waiter notification.

function release(sem) {
  if (sem.waiters.length > 0) {
    const next = sem.waiters.shift();
    next.resolve(); // direct handoff
  } else {
    sem.active--;   // no waiters, free the slot
  }
}

Dynamic Concurrency

The maxConcurrency is updated every time getSemaphore() is called, so changes to model_configs.max_concurrency via the admin API take effect on the next request without restart.

Queue Statistics

getQueueStats() returns the state of all semaphores for the /metrics endpoint:

Pricing

Cost calculation is handled by calculateCost() in cost-calculator.mjs. Two pricing models are supported:

Token-based Pricing

Most direct API providers (OpenAI, Anthropic, Google) charge per token:

input_cost  = (prompt_tokens / 1,000,000) * input_price
output_cost = (completion_tokens / 1,000,000) * output_price
total_cost  = input_cost + output_cost

Prices are stored in the model_configs table as USD per 1M tokens. If the database has no pricing and the model is token-priced, OpenRouter's API is queried as a fallback.

Per-request Pricing

Subscription-based providers (Copilot, Kiro) charge a flat rate per request regardless of token count:

input_cost  = 0
output_cost = 0
total_cost  = request_cost

Request costs are set by the migration in init.mjs based on model tier:
