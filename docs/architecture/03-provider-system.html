Soul Gateway Architecture
Part 3

Provider System

How Soul Gateway authenticates with upstream LLM providers: three auth types, OAuth adapters for five services, multi-account credential pooling, token refresh, format converters, and auto-provisioning.

Three Auth Types

Each entry in provider_configs has an auth_type field that determines how the gateway authenticates with the upstream provider:

Auth TypeMechanismExamples
api_key Static API key stored encrypted in the encrypted_api_key column of provider_configs. Decrypted at dispatch time and sent as a Bearer token. OpenAI, Anthropic (direct), Google, OpenRouter
managed OAuth credentials stored on disk in the credential store (/shared/soul-gateway/providers/). Tokens are refreshed automatically. The encrypted_api_key column is NULL. Copilot, Kiro, Codex, Gemini, Anthropic (OAuth)
internal Used for internal services that run alongside Soul Gateway (e.g., Search Gateway). No external authentication needed beyond what the service itself handles. Search Gateway

Auth Manager

The auth-manager.mjs module is the central coordinator for managed-auth providers. It maintains a registry of adapter modules and orchestrates credential retrieval, token refresh, account rotation, and auto-provisioning.

Key Functions

  • registerAdapter(adapter) -- Registers an OAuth adapter (called during startup for each provider)
  • getCredentials(providerName) -- Returns { token, headers, formatConverter } for the active account, refreshing if needed
  • startAuth(providerName) -- Initiates an OAuth flow (device flow or PKCE)
  • pollAuth(providerName) -- Polls device flow for completion
  • handlePKCECallback(providerName, code, state) -- Handles OAuth redirect callback
  • rotateAccount(providerName) -- Marks current account as quota-exhausted and switches to next
  • reconcileProviders() -- Startup reconciliation: ensures DB rows exist for all credential-bearing providers

Credential Store

The credential store (credential-store.mjs) persists OAuth tokens as JSON files on disk. The default base directory is /shared/soul-gateway/providers/ (overridable via CREDENTIAL_STORE_PATH env var).

File Structure

OAuth Adapters

Each managed-auth provider has an adapter module in providers/adapters/. Adapters implement a standard interface:

// Adapter interface
{
  name: string,           // Provider name (matches provider_configs.name)
  authType: string,       // 'device-flow' or 'pkce'
  refreshMarginMs: number,// Refresh N ms before expiry (default 60000)
  formatConverter: string,// Name of format converter module (or null)
  providerTemplate: {},   // Template for auto-provisioning provider_configs row
  knownModels: string[],  // Models to auto-create on first login

  // Device flow adapters
  startAuth(): { deviceCode, userCode, verificationUri, interval },
  pollForToken(deviceCode, interval): credentials,

  // PKCE flow adapters
  startAuth(): { authUrl },
  exchangeCode(code, state): credentials,

  // Common
  refreshToken(account): refreshedCredentials,
  getHeaders(account): { Authorization, ... },
}

Adapter Details

AdapterAuth FlowFormat ConverterNotes
Copilot Device Flow (GitHub) copilot-responses Authenticates via github.com/login/device. Gets a Copilot token from GitHub API. Token auto-refreshes. Subscription-billed.
Kiro Cognito + PKCE kiro-eventstream AWS Cognito-based auth with PKCE code challenge. Uses a local callback server for the redirect.
Codex PKCE (ChatGPT) copilot-responses Authenticates against chatgpt.com OAuth. Uses chatgpt.com/backend-api/codex/responses endpoint.
Gemini Device Flow (Google) None (native OpenAI-compatible) Google OAuth device flow. Limited device flow endpoint.
Anthropic PKCE anthropic-messages Anthropic OAuth for free-tier models (Haiku). Paid models require separate routing.
Search Internal search Not OAuth -- adapter for the Search Gateway internal service.

Multi-Account Credential Pooling

Each provider supports multiple accounts (e.g., multiple GitHub accounts for Copilot). The credential store tracks them as account-0.json, account-1.json, etc. The state.json file records which account index is currently active.

Account Rotation

When an account hits its quota (e.g., Copilot premium request limit), the rotateAccount() function:

  1. Marks the current account as quotaExhausted = true
  2. Sets quotaResetAt to midnight UTC of the next day
  3. Finds the next non-exhausted account and switches to it
  4. Returns false if all accounts are exhausted

Account rotation is triggered during the retry loop in retry.mjs when a managed provider returns HTTP 402 (payment required):

if (err.dbConfig?.auth_type === 'managed' &&
    (err.errorClassification?.type === 'payment_required' || err.status === 402)) {
  const rotated = await authManager.rotateAccount(err.dbConfig.name);
  if (rotated) continue; // retry with new account
  throw ... // all accounts exhausted
}

Token Refresh Background Loop

The startRefreshLoop() function runs every 60 seconds (configurable). For each registered adapter and each account, it:

  1. Quota reset: If an account's quotaResetAt has passed, clears the quotaExhausted flag
  2. Token refresh: If expiresAt is within the adapter's refreshMarginMs (default 60s) and the account doesn't need re-auth, calls adapter.refreshToken(account)

Token refresh is coalesced: concurrent refresh requests for the same provider+account index are deduplicated using a refreshInProgress Map of Promises. If refresh fails, the account is marked with needsReauth = true so it is skipped until the user re-authenticates.

Format Converters

Not all upstream providers speak OpenAI's chat completion format natively. Format converters translate between the gateway's OpenAI-compatible interface and provider-specific protocols:

ConverterFileUsed ByDescription
anthropic-messages anthropic-messages.mjs Anthropic OAuth Converts OpenAI messages to Anthropic Messages API format and vice versa
copilot-responses copilot-responses.mjs Copilot, Codex Converts to/from GitHub Copilot's responses API format
kiro-eventstream kiro-eventstream.mjs Kiro Handles Kiro's event-stream response format
search search.mjs Search Gateway Converts search results to chat completion format

Format converters are referenced by name in the adapter's formatConverter field and loaded during upstream dispatch.

Auto-Provisioning

When a user completes an OAuth login for the first time, the gateway automatically creates the necessary database records. The autoProvision() function (called after every successful auth and at startup via reconcileProviders()):

  1. Provider creation: Checks if a provider_configs row exists for the provider name. If not, creates one from the adapter's providerTemplate (which includes display_name, protocol, base_url, billing_type, auth_type).
  2. Model creation: If the adapter has a knownModels array, creates a model_configs row for each model using upsertModel(). Model names follow the axl/<provider>/<model> convention via buildModelName().

Both steps are idempotent -- safe to call on every login and every startup. The reconcileProviders() function iterates all registered adapters, checks if any have credentials on disk, and runs auto-provision for those that do.


  OAuth Login Complete
        |
  autoProvision(providerName, adapter)
        |
  +-----+-------------------------------------------+
  |                                                   |
  1. getProviderByName(name)                          |
     - exists? skip                                   |
     - missing? createProvider(template)              |
  |                                                   |
  2. for each knownModel:                             |
       upsertModel({                                  |
         name: buildModelName(provider, model),       |
         provider_key: provider,                      |
         provider_model: model,                       |
       })                                             |
  +---------------------------------------------------+
