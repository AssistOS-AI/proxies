Soul Gateway Architecture
Architecture

System Architecture

Deep-dive into every layer of Soul Gateway: how requests flow from ingress to response, how models are routed and retried, how the database is partitioned, and how the middleware plugin system ties it all together.

Part 1

Request Pipeline

The full lifecycle of a /v1/chat/completions request: authentication, body parsing, model routing, middleware execution, dispatch with retry, streaming, cost calculation, logging, and real-time broadcast.

 Part 2

Database Schema

PostgreSQL tables powering the gateway: API keys, model configs, provider configs, call logs with monthly partitioning, middleware registry, rate-limit state, and the incremental migration strategy.

 Part 3

Provider System

How Soul Gateway authenticates with upstream LLM providers: API key, managed OAuth, and internal auth types. OAuth adapters, multi-account credential pooling, token refresh, format converters, and auto-provisioning.

 Part 4

Middleware Framework

The pluggable middleware system: file-based discovery, the before/after hook interface, context object anatomy, execution ordering, abort mechanics, settings merging, and the 12 built-in middlewares.

 Part 5

Streaming & WebSockets

SSE response streaming from upstream providers to clients, the raw RFC 6455 WebSocket implementation, real-time log broadcasting over WebSocket and SSE, ping/pong heartbeats, and frame encoding.

 Part 6

Model Routing

Model resolution algorithm: direct lookup, tier-based fallback chains with cycle detection, cooldown system for transient errors, per-model concurrency queues, and token/request-based pricing.
