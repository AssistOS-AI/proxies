# OpenCode Zen + OpenCode Go Builtin Providers Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add OpenCode Zen and OpenCode Go to the builtin provider preset catalog so they appear in the dashboard "Add Provider" dropdown as OpenAI-compatible vendors.

**Architecture:** Both endpoints are OpenAI-compatible, so they are added as pure-data presets in `PROVIDER_PRESETS` that reuse the already-loaded `openai-api` backend via the shared `OPENAI_COMPAT_DEFAULTS` bundle (identical pattern to NVIDIA/Groq/OpenRouter). No backend code, no `achillesAgentLib` change. The single existing test that asserts the preset catalog is updated in the same task.

**Tech Stack:** Node.js ES modules, `node:test` runner.

## Global Constraints

- ES modules with `import` / `export`; four-space indentation for JavaScript.
- Verified base URLs (do not alter): OpenCode Zen = `https://opencode.ai/zen/v1`; OpenCode Go = `https://opencode.ai/zen/go/v1` (NOT `https://opencode.ai/go/v1`).
- Preset keys/display names: `opencode-zen` / "OpenCode Zen", `opencode-go` / "OpenCode Go".
- Scope is preset + test only. Do not edit deploy workflows, `API_KEYS.md`, backend modules, or `achillesAgentLib`.
- Run all commands from the `soul-gateway/` directory.
- Commit messages must contain no AI/coding-agent attribution (no `Co-Authored-By`, no "Generated by"), per `soul-gateway/CLAUDE.md`.
