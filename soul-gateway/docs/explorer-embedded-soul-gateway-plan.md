# Soul Gateway Embedded Explorer Plan

Status: analysis-backed plan. No implementation has been done in this document.

## Summary

Soul Gateway is already a Ploinky agent. The missing work is to add an embedded operating mode so it can start as an Explorer dependency, use Explorer's authenticated admin session, seed a local OpenAI-compatible LLM provider by default, and still keep the current standalone `soul.axiologic.dev` deployment unchanged.

The clean target is two manifest-selected behaviors:

- Standalone mode: the current production shape, port-bound on `8042`, dashboard password/API-key based, Postgres-backed, externally reachable through the existing deployment.
- Embedded Explorer mode: started by `AssistOSExplorer/explorer`, reachable through Ploinky HTTP service routes, protected by Explorer/Ploinky auth, and pre-wired with a workspace-internal derived API key plus a local LLM provider.

The profile names `embedded` and `standalone` are the selected operator language. Ploinky currently only accepts global active profile names `default`, `dev`, `qa`, and `prod`, so supporting semantic manifest profiles is part of this plan.

## Goals

1. Deploying `AssistOSExplorer/explorer` starts Soul Gateway automatically as a sibling Ploinky agent.
2. The embedded deployment needs no manual Soul Gateway API-key generation.
3. The embedded deployment has a default local LLM provider ready from manifest/profile defaults.
4. Explorer admins can open Soul Gateway management without a separate Soul Gateway login.
5. Explorer admins can add API keys for other LLM providers from Settings -> Plugins -> Soul Gateway Settings.
6. Existing standalone Soul Gateway deployments, including `soul.axiologic.dev`, keep their current behavior.

## What Exists Today

### Soul Gateway

- `proxies/soul-gateway/manifest.json` already defines a Ploinky agent with `container: "node:20-slim"`, `agent: "bash /code/startup-v2.sh"`, `cli: "bash /code/cli.sh"`, `enable: ["postgres"]`, and one `default` profile.
- The current `default` profile publishes `0.0.0.0:8042:8042` and uses `install-v2.sh`.
- The manifest does not declare `httpServices`, so Ploinky's router does not currently expose it under `/services/...`.
- `startup-v2.sh` prepares `/app/src`, syncs or installs `achillesAgentLib`, sets `DATA_DIR` and `CREDENTIALS_DIR`, constructs `DATABASE_URL` from `PG*` vars when needed, defaults `PORT` to `8042`, and starts `node src/index.mjs`.
- Runtime API auth is bearer API-key based through the `api_keys` table. Dashboard auth is `soul_session` cookie based and requires `DASHBOARD_PASSWORD` unless the code path is changed.
- Provider setup is manual today through `/management/providers` and `/management/keys`.

### Ploinky

- `ploinky start <agent>` resolves the agent manifest, enables dependencies from manifest `enable[]`, starts dependency agents, then starts the static agent and router.
- `workspaceDependencyGraph.js` reads both top-level `enable[]` and `profiles[activeProfile].enable[]`, where `activeProfile` comes from `.ploinky/profile`.
- `profileService.js` currently restricts the active workspace profile to `default`, `dev`, `qa`, or `prod`. A manifest can contain other profile names, but the normal `ploinky profile embedded` path rejects them.
- Docker services with no manifest `ports` get a generated localhost host port mapped to container port `7000`. Therefore an embedded Soul Gateway with no `ports` must listen on `PORT=7000`, or Ploinky must learn how to publish an arbitrary container port with a random localhost host port.
- Ploinky injects `PLOINKY_ROUTER_URL`, `PLOINKY_ROUTER_HOST`, `PLOINKY_ROUTER_PORT`, `PLOINKY_AGENT_PRINCIPAL`, and `PLOINKY_DERIVED_MASTER_KEY` into agents.
- Manifest env entries support `derive: "derived-master"`, including cross-agent derivation fields such as `deriveRepoName`, `deriveAgentName`, and `deriveName`. This is the right mechanism for a shared workspace-internal Soul Gateway API key.
- HTTP service routes are manifest-declared through `httpServices`. Protected services receive router-authenticated identity in `x-ploinky-auth-info`; caller-supplied Ploinky identity headers are stripped and regenerated by the router.
- The auth info header can include a router-issued invocation token for tool `__http_service__`. Agents can verify invocation JWTs with `PLOINKY_DERIVED_MASTER_KEY` using the existing Ploinky invocation verifier.

### Explorer

- `AssistOSExplorer/explorer/manifest.json` currently treats Soul Gateway as external: it declares `SOUL_GATEWAY_API_KEY` and `SOUL_GATEWAY_BASE_URL`, but does not enable `proxies/soul-gateway`.
- `AssistOSExplorer/llmAssistant/manifest.json` also declares `SOUL_GATEWAY_API_KEY` and `SOUL_GATEWAY_BASE_URL`.
- The Explorer deployment workflow writes optional `SOUL_GATEWAY_API_KEY` and `SOUL_GATEWAY_BASE_URL` values into Ploinky vars, then starts `AchillesIDE/explorer`.
- The Explorer runtime plugin system discovers `IDE-plugins` directories from the workspace and installed repos. The Settings modal already lists plugins in the built-in Plugins tab and opens a plugin-specific settings modal when a plugin config declares `"settings": "<component-name>"`.
- The Settings modal has an admin access concept for the built-in Administration tab, but runtime plugin entries do not yet propagate an `adminOnly` flag into the plugin settings list.

## Target Architecture

### Standalone Mode

Standalone mode remains the current production mode:

- Ploinky starts `soul-gateway` directly.
- `PORT=8042`.
- `ports: ["0.0.0.0:8042:8042"]`.
- Postgres remains enabled.
- Dashboard login uses `DASHBOARD_PASSWORD`.
- API clients use DB-backed API keys issued by an admin.
- `soul.axiologic.dev` and its deploy workflow keep working.

### Embedded Explorer Mode

Embedded mode runs Soul Gateway as an Explorer dependency:

- Explorer manifest enables `proxies/soul-gateway`.
- Soul Gateway listens on `PORT=7000` so Ploinky can publish a random localhost-only host port when no explicit `ports` are declared.
- Browser and sibling agents reach the gateway through the router, for example `${PLOINKY_ROUTER_URL}/services/soul-gateway/v1`.
- Management endpoints are protected Ploinky HTTP services, not a second login surface.
- If the Explorer user has an admin role, Soul Gateway management accepts that identity without a Soul Gateway password.
- Explorer and llmAssistant get a derived `SOUL_GATEWAY_API_KEY` that matches Soul Gateway's own embedded default key. No manual API-key generation is needed.
- On first start, Soul Gateway seeds a `local-llm` provider that points at the default OpenAI-compatible local LLM deployment.
- Admins can later add external provider keys through Explorer Settings.

### Shared Implementation Principle

Embedded and standalone modes should share the same source tree, `src/index.mjs`, request pipeline, provider runtime, and management routes. Mode-specific behavior should be selected by manifest profile/env and small gated branches, not by forking the service.

## Profile Plan

Decision: implement semantic `embedded` and `standalone` Ploinky profiles. Do not use `prod` as the embedded overlay.

Add support for manifest-declared semantic profiles so this can be expressed directly:

```jsonc
{
  "profiles": {
    "default": { "mode": "standalone-compatible base" },
    "standalone": { "mode": "explicit standalone alias" },
    "embedded": { "mode": "Explorer dependency mode" }
  }
}
```

Required Ploinky changes:

- Allow `ploinky profile <name>` to accept names declared by the static agent or by enabled dependency manifests, not only `default/dev/qa/prod`.
- Ensure dependency graph profile lookup uses the selected semantic profile.
- Add a dependency-local profile override in `enable` object entries and persist it in `agents.json`, so Explorer can run with its normal profile while Soul Gateway runs with `embedded`.

## Soul Gateway Manifest Plan

Keep the existing `default` profile stable for standalone production and add explicit `standalone` and `embedded` profiles. The `standalone` profile should preserve the current production behavior. The `embedded` profile has these important differences:

```jsonc
{
  "env": {
    "PORT": { "required": false, "default": "7000" },
    "SOUL_GATEWAY_MODE": { "required": false, "default": "embedded" },
    "TRUST_PLOINKY_ROUTER_AUTH": { "required": false, "default": "true" },
    "ALLOW_UNAUTHENTICATED": { "required": false, "default": "false" },
    "LOCAL_LLM_BASE_URL": {
      "required": false,
      "default": "http://host.containers.internal:11434/v1",
      "description": "Default OpenAI-compatible endpoint for the local LLM deployment. Override only when the local LLM is hosted elsewhere."
    },
    "LOCAL_LLM_MODEL": {
      "required": false,
      "default": "gemma4:e2b"
    },
    "ENCRYPTION_KEY": {
      "required": true,
      "derive": "derived-master",
      "deriveName": "soul-gateway-encryption-key"
    },
    "ADMIN_SESSION_SIGNING_KEY": {
      "required": true,
      "derive": "derived-master",
      "deriveName": "soul-gateway-admin-session"
    },
    "SOUL_GATEWAY_API_KEY": {
      "required": true,
      "derive": "derived-master",
      "deriveName": "workspace-default-api-key"
    },
    "DASHBOARD_PASSWORD": {
      "required": false,
      "default": ""
    },
    "OAUTH_ADAPTERS_ENABLED": {
      "required": false,
      "default": ""
    },
    "TOKEN_REFRESH_INTERVAL_MS": {
      "required": false,
      "default": "0"
    },
    "PRICING_REFRESH_INTERVAL_MS": {
      "required": false,
      "default": "0"
    }
  },
  "install": "bash /code/install-v2.sh"
}
```

Declare `ports: []` in embedded mode. This clears the standalone profile's `8042` binding, lets Ploinky map a random localhost host port to container port `7000`, and lets the router use that host port for HTTP service proxying.

Add root-level `httpServices`:

```jsonc
[
  {
    "externalPrefix": "/services/soul-gateway/v1/",
    "internalPrefix": "/v1/",
    "auth": "protected",
    "notFoundMessage": "Soul Gateway route not found."
  },
  {
    "externalPrefix": "/services/soul-gateway/management/",
    "internalPrefix": "/management/",
    "auth": "protected",
    "notFoundMessage": "Soul Gateway management route not found."
  },
  {
    "externalPrefix": "/public-services/soul-gateway-health/",
    "internalPrefix": "/healthz/",
    "auth": "none",
    "notFoundMessage": "Soul Gateway health route not found."
  }
]
```

Keep the public health route for deployment smoke checks. Add a `/healthz/` alias in Soul Gateway because Ploinky normalizes service prefixes with trailing slashes.

For storage, keep Postgres in embedded mode. Soul Gateway's DAOs and migrations are already Postgres-oriented, and preserving that avoids a large schema-porting effort. SQLite is not part of this plan.

### Embedded Runtime Defaults

Embedded mode should set `DASHBOARD_PASSWORD=""` because management access comes from verified Explorer/Ploinky admin identity. OAuth adapters should be disabled by default with `OAUTH_ADAPTERS_ENABLED=""`; external API-key providers remain supported through the settings plugin. Token refresh and pricing refresh intervals should default to `0` in embedded mode, and the scheduler should explicitly treat `0` as disabled before this profile relies on those values.

## Auth Plan

### Explorer Admin SSO

Add a router-auth module in Soul Gateway:

- Only runs when `TRUST_PLOINKY_ROUTER_AUTH=true` and `SOUL_GATEWAY_MODE=embedded`.
- Reads `x-ploinky-auth-info`.
- Requires a user object with an admin role.
- Requires and verifies the embedded `invocationToken` as a Ploinky invocation JWT for audience `agent:proxies/soul-gateway` and tool `__http_service__`.
- Rejects requests without a valid invocation token, even if they contain an admin-looking header.

Then update management auth so `requireAdmin()` tries router admin SSO first and falls back to the current `soul_session` cookie/password flow.

Also update Explorer's `pwd.users` entries to make roles explicit:

```jsonc
{
  "username": "admin",
  "password": "admin",
  "name": "Admin",
  "roles": ["admin"]
}
```

If legacy deployments lack explicit roles, a short transition fallback can accept `username === "admin"` only in embedded mode, but the durable contract should be role based.

### Derived API Key For Agent Calls

Add an embedded API-key shortcut:

- If `SOUL_GATEWAY_MODE=embedded` and bearer token equals `SOUL_GATEWAY_API_KEY`, authorize a `workspace-default` key.
- When Postgres is configured, idempotently persist the embedded key in `api_keys` so request sessions, budgets, and audit rows use a real FK-compatible key id.
- When no database is configured, fall back to an in-memory synthetic key record.
- Mark metadata so calls made with the embedded workspace key are distinguishable.
- Keep the DB-backed key path unchanged for standalone and for extra API keys generated later.

Explorer and llmAssistant should derive the same key by referencing Soul Gateway's derivation identity:

```jsonc
{
  "name": "SOUL_GATEWAY_API_KEY",
  "required": true,
  "derive": "derived-master",
  "deriveRepoName": "proxies",
  "deriveAgentName": "soul-gateway",
  "deriveName": "workspace-default-api-key"
}
```

## Default Local LLM Plan

Embedded mode should not require the user to create a provider manually. Add an idempotent bootstrap step:

1. If not embedded, return immediately.
2. If a provider already exists, do nothing.
3. Read `LOCAL_LLM_BASE_URL` and `LOCAL_LLM_MODEL`.
4. Create provider `local-llm` with adapter `openai-api`, `authStrategy: "none"`, and the configured base URL. The embedded profile default is `http://host.containers.internal:11434/v1`, aligned with the existing `basic/ollama` Ploinky agent.
5. Attempt model discovery.
6. If discovery fails or the server has no `/models`, create a fallback model row from `LOCAL_LLM_MODEL` when it is not `auto`.
7. If the default local LLM endpoint is unreachable, boot diagnostics may return a clear 503, but embedded deployment verification should treat that as a failed deployment. The default target is expected to exist.

This requires one backend adjustment: the `openai-api` backend must allow lifecycle/model calls with no credential lease when the provider auth strategy is `none`. Today model discovery expects a token.

The local LLM URL is non-secret topology. It should have a manifest default and can be overridden by:

- A Ploinky var such as `LOCAL_LLM_BASE_URL`.
- The new Explorer Settings page.

## CORS Plan

No Soul Gateway CORS changes are required for embedded mode. Browser calls go to the Ploinky router origin and are proxied through protected HTTP service routes. Standalone deployments can keep relying on their existing public deployment and proxy configuration.

## Explorer Integration Plan

### Manifest Wiring

Update `AssistOSExplorer/explorer/manifest.json`:

- Add `proxies/soul-gateway` to `enable[]` once the Explorer deployment installs/enables the `proxies` repo.
- Convert `SOUL_GATEWAY_API_KEY` to the derived-master spec shown above.
- Stop requiring `SOUL_GATEWAY_BASE_URL` for embedded operation.
- Add explicit roles to the admin `pwd` user.

Update `AssistOSExplorer/llmAssistant/manifest.json`:

- Convert `SOUL_GATEWAY_API_KEY` to the same derived-master spec.
- Make `SOUL_GATEWAY_BASE_URL` optional.

Update LLM configuration resolution:

- If `SOUL_GATEWAY_BASE_URL` is set, keep using it. This preserves current external gateway deployments.
- If it is not set and `PLOINKY_ROUTER_URL` exists, default to `${PLOINKY_ROUTER_URL}/services/soul-gateway/v1`.
- Keep all request-time LLM inference through `achillesAgentLib`.

This avoids hardcoding a fake `http://router/...` hostname. Ploinky already injects the correct host/port URL for the active runtime.

### Deployment Workflow

Update `AssistOSExplorer/.github/workflows/deploy-skills-explorer.yml`:

- Add the `proxies` repo to the workspace with the desired branch.
- Enable the `proxies` repo.
- Let Explorer's manifest dependency graph start Soul Gateway.
- Stop requiring `SOUL_GATEWAY_API_KEY` and `SOUL_GATEWAY_BASE_URL` secrets for the embedded default.
- Keep honoring explicit `SOUL_GATEWAY_BASE_URL` and `SOUL_GATEWAY_API_KEY` vars when the operator wants Explorer to use an external standalone gateway.
- Ensure the default local LLM deployment is available to the Explorer host. The existing `basic/ollama` agent is the default local option; external remote local LLM deployments can override `LOCAL_LLM_BASE_URL` / `LOCAL_LLM_MODEL`.

Keep the standalone Soul Gateway workflow behavior unchanged. If the workflow begins selecting profiles explicitly, set it to `standalone`.

## Explorer Settings Plugin Plan

Create a Soul Gateway runtime plugin:

```text
proxies/soul-gateway/IDE-plugins/soul-gateway-settings/
  config.json
  soul-gateway-settings.html
  soul-gateway-settings.js
  soul-gateway-settings.css
```

Suggested `config.json`:

```json
{
  "pluginCategory": "application",
  "id": "soul-gateway",
  "component": "soul-gateway-settings",
  "settings": "soul-gateway-settings",
  "label": "Soul Gateway",
  "tooltip": "Configure Soul Gateway providers",
  "presenter": "SoulGatewaySettings",
  "type": "global",
  "adminOnly": true
}
```

With the current Explorer UI, this appears in Settings -> Plugins with a Settings button. That is the target UX for this plan; do not add a first-class Soul Gateway tab in the initial implementation.

The settings component should call protected router routes:

- `GET /services/soul-gateway/management/providers/templates`
- `GET /services/soul-gateway/management/providers`
- `POST /services/soul-gateway/management/providers`
- `PATCH /services/soul-gateway/management/providers/:id`
- `POST /services/soul-gateway/management/providers/:id/keys` or the existing key/account route shape
- `GET /services/soul-gateway/management/models`

The UI should support:

- Local LLM status and base URL.
- Model discovery/test connection.
- External provider add/edit/remove.
- Write-only API-key input fields.
- API-key list/generate/revoke flows for DB-backed keys, excluding the synthetic `workspace-default` embedded key from display and generation flows.
- Provider/model status errors.

Explorer changes needed:

- Preserve `adminOnly` from plugin config when flattening plugin entries for the Settings modal.
- Hide or disable admin-only plugin settings for non-admin users.
- Reuse the existing admin access probe or add a small current-user/role endpoint.

Soul Gateway should enforce admin server-side regardless of UI hiding.

## Backward Compatibility

- Standalone `soul.axiologic.dev` remains Postgres-backed and port-bound.
- Existing DB-backed API keys remain valid.
- Existing explicit `SOUL_GATEWAY_BASE_URL` and `SOUL_GATEWAY_API_KEY` vars continue to override embedded defaults.
- New router-auth behavior is gated by embedded env and invocation-token verification.
- Adding `httpServices` is inert for direct standalone access unless the service is run behind a Ploinky router.
- Rotating `PLOINKY_MASTER_KEY` changes the derived embedded API key, but Explorer and Soul Gateway derive it in sync.

## Implementation Phases

### Phase 1: Ploinky Profile And Routing Support

- Add semantic `embedded` and `standalone` profile support to Ploinky profile validation, dependency graph profile lookup, and tests.
- Add Soul Gateway `httpServices`.
- Add an embedded profile with `PORT=7000`.
- Verify the router can proxy `/services/soul-gateway/v1/...` to the generated localhost port.

### Phase 2: Embedded Auth

- Add Soul Gateway router SSO verification.
- Reuse Ploinky invocation JWT verification rather than trusting a plain header; verify audience, tool, signed `invocationBody` hash, and replay id.
- Add embedded derived API-key shortcut with DB persistence when Postgres is configured.
- Add unit tests for standalone unchanged behavior, valid admin SSO, non-admin rejection, missing invocation-token rejection, body mismatch, replay rejection, and embedded workspace API-key auth.

### Phase 3: Local LLM Bootstrap

- Add idempotent embedded provider bootstrap.
- Add an `authStrategy: "none"` path for OpenAI-compatible provider discovery/execution.
- Add tests with a mock OpenAI-compatible server.
- Verify the default `LOCAL_LLM_BASE_URL` reaches the local LLM deployment; unreachable local LLM is an embedded deployment failure.

### Phase 4: Explorer Wiring

- Add `proxies` repo to Explorer deploy workflow.
- Enable Soul Gateway as an Explorer dependency.
- Derive `SOUL_GATEWAY_API_KEY` in Explorer and llmAssistant.
- Compute default gateway URL from `PLOINKY_ROUTER_URL` when no explicit external URL is configured.
- Add explicit admin role to Explorer's local admin user.

### Phase 5: Settings Plugin

- Add the Soul Gateway settings plugin under `IDE-plugins`.
- Extend Explorer Settings plugin metadata handling for `adminOnly`.
- Build provider/key management UI against the protected management routes.
- Add UI and server tests for admin-only visibility and provider CRUD.

### Phase 6: Deployment Verification

- Standalone smoke: deploy or start Soul Gateway in current mode, confirm `https://soul.axiologic.dev/healthz` and `http://localhost:8042/healthz`.
- Embedded smoke order:
  1. Confirm router health on the configured router port.
  2. Confirm `curl http://localhost:<routerPort>/public-services/soul-gateway-health/` returns minimal healthy JSON.
  3. Confirm `ploinky status` shows Explorer, Soul Gateway, Postgres, and the default local LLM dependency running.
  4. Log in to Explorer as admin.
  5. Open Settings -> Plugins -> Soul Gateway Settings.
  6. Confirm non-admin users cannot open or use Soul Gateway management.
  7. Trigger one llmAssistant request through Soul Gateway and the default local LLM provider.

### Phase 7: Documentation

- Add `proxies/soul-gateway/docs/specs/DS016-embedded-mode.md` for the embedded mode contract.
- Update `proxies/soul-gateway/docs/specs/DS013-configuration-deployment.md` with the `embedded` and `standalone` profile behavior.
- Update `proxies/soul-gateway/CLAUDE.md` and `proxies/CLAUDE.md` with the embedded versus standalone framing.
- Update `AssistOSExplorer/CLAUDE.md` and `AssistOSExplorer/explorer/CLAUDE.md` to mention Soul Gateway as an enabled sibling agent in the embedded topology.

## Decisions

1. Profile mechanism: implement semantic `embedded` and `standalone` Ploinky profiles.
2. Storage: keep Postgres in embedded mode.
3. Local LLM default: set `LOCAL_LLM_BASE_URL` by default in the embedded profile, using `http://host.containers.internal:11434/v1` unless the deployment selects a different local LLM endpoint.
4. Settings UX: use Settings -> Plugins -> Soul Gateway Settings.
5. Health route: expose a minimal `/public-services/soul-gateway-health/` route for deployment smoke checks.

## Out Of Scope

- Replacing `soul.axiologic.dev` with embedded Explorer mode.
- Cross-workspace federation.
- OAuth provider callback support in embedded mode.
- Non-admin provider management.
- Migrating standalone production storage away from Postgres.
