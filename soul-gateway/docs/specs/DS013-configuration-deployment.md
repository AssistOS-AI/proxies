# DS013 -- Configuration & Deployment

## Summary

This specification covers all configuration surfaces for Soul Gateway -- environment variables (from manifest and application config), the Ploinky manifest structure, and the full deployment lifecycle including install, startup, production deploy, health checks, and graceful shutdown.

## Environment Variables

### Manifest Variables

These variables are declared in `manifest.json` under `profiles.default.env`. Ploinky injects them into the container at startup. All are optional.

| Variable | Default | Description |
|----------|---------|-------------|
| `PORT` | `8042` | HTTP server listen port. |
| `UPSTREAM_URL` | `https://proxy.axiologic.dev` | CLIProxyAPI upstream URL used as the default backend for the axiologic provider. |
| `PGHOST` | `host.containers.internal` | PostgreSQL host. Uses the Podman host-gateway alias so the container can reach the host's PostgreSQL. |
| `PGPORT` | `5432` | PostgreSQL port. |
| `PGUSER` | `postgres` | PostgreSQL user. |
| `PGPASSWORD` | `postgres` | PostgreSQL password. |
| `PGDATABASE` | `postgres` | PostgreSQL database name. |
| `ENCRYPTION_KEY` | *(empty)* | AES-256 hex key for encrypting API keys at rest. Auto-generated by `install.sh` if empty and persisted to `/shared/soul-gateway/config/encryption.key`. |
| `WORKSPACE_PATH` | *(empty)* | Absolute path to the Ploinky workspace. Fallback to locate source code when `/code` is not mounted. |
| `DEFAULT_PROXY_API_KEY` | *(empty)* | API key for authenticating with the CLIProxyAPI upstream. Inherited by `AXIOLOGIC_PROXY_API_KEY` in `startup.sh` if not set separately. |
| `DASHBOARD_PASSWORD` | *(empty)* | Password protecting the web dashboard. When empty, the dashboard has no authentication. |
| `AXIOLOGIC_PROXY_API_KEY` | *(empty)* | API key for the axiologic proxy provider (CLIProxyAPI). Falls back to `DEFAULT_PROXY_API_KEY`. |
| `OPENROUTER_API_KEY` | *(empty)* | API key for the OpenRouter provider. |
| `COPILOT_API_KEY` | `no-auth-needed` | Dummy key for the Copilot provider. The copilot-gateway sidecar handles its own authentication via GitHub device flow. |
| `KIRO_PROXY_API_KEY` | `no-auth-needed` | Dummy key for the Kiro provider. The kiro-gateway sidecar handles its own authentication. |
| `BRAVE_API_KEY` | *(empty)* | API key for Brave web search provider. |
| `SERPER_API_KEY` | *(empty)* | API key for Serper (Google) web search provider. |
| `JINA_API_KEY` | *(empty)* | API key for Jina web search provider (optional). |

> **Search provider API keys:** `TAVILY_API_KEY`, `EXA_API_KEY`, and `GEMINI_SEARCH_API_KEY` are no longer set via environment variables in the deploy workflow. Configure these keys through the Soul Gateway dashboard UI instead. The remaining search provider keys (`BRAVE_API_KEY`, `SERPER_API_KEY`, `JINA_API_KEY`) can still be set as environment variables.

### Application Defaults (config.mjs)

These values are defined in `src/config.mjs`. Some can be overridden by environment variables (noted below); others are compile-time constants changed by editing the source.

#### Dashboard & Database

| Config Key | Default | Env Override | Description |
|------------|---------|--------------|-------------|
| `port` | `8042` | `PORT` | HTTP server listen port. |
| `dashboardPassword` | *(empty)* | `DASHBOARD_PASSWORD` | Dashboard authentication password. |
| `pgSchema` | `soul_gateway` | -- | PostgreSQL schema name. All tables are created under this schema. |
| `encryptionKey` | *(empty)* | `ENCRYPTION_KEY` | AES-256 hex key. Loaded from shared storage by `startup.sh` if not set. |
| `llmConfigPath` | *(empty)* | `LLM_CONFIG_PATH` | Optional override path to the achillesAgentLib `LLMConfig.json` file. |

#### Rate Limiting

| Config Key | Default | Description |
|------------|---------|-------------|
| `defaultRpmLimit` | `60` | Default requests-per-minute limit for new API keys. |
| `defaultTpmLimit` | `100,000` | Default tokens-per-minute limit for new API keys. |

#### Retry Behavior

| Config Key | Default | Description |
|------------|---------|-------------|
| `maxRetries` | `3` | Maximum number of retry attempts per request. |
| `initialDelayMs` | `1,000` | Initial delay in ms before the first retry. |
| `backoffMultiplier` | `2` | Exponential backoff multiplier applied to each subsequent retry. |
| `maxDelayMs` | `30,000` | Maximum delay cap in ms for any single retry wait. |
| `jitterPercent` | `20` | Random jitter percentage applied to retry delays to avoid thundering herds. |

#### Alert Thresholds

| Config Key | Default | Description |
|------------|---------|-------------|
| `slowRequestMs` | `30,000` | Requests exceeding this latency are flagged as slow in logs. |
| `largePromptTokens` | `50,000` | Prompts exceeding this token count trigger a large-prompt alert. |

#### Model Cooldown

| Config Key | Default | Env Override | Description |
|------------|---------|--------------|-------------|
| `cooldownDurationMs` | `3,600,000` (1 hour) | `COOLDOWN_DURATION_MS` | How long a model stays in cooldown after a trigger error (quota/payment). |
| `maxModelRetries` | `5` | `MAX_MODEL_RETRIES` | Maximum retries before a model enters cooldown. |
| `cooldownTriggers` | `['payment_required', 'rate_limit_error']` | -- | Error types that trigger model cooldown. A model receiving these errors is temporarily skipped in routing. |

#### Logging & Sessions

| Config Key | Default | Description |
|------------|---------|-------------|
| `retentionDays` | `90` | Number of days to retain call log records before cleanup. |
| `sessionTimeoutMs` | `1,800,000` (30 min) | Gap in ms between requests from the same key+agent that triggers a new session. |

### Provider API Key Cascade

The `startup.sh` script applies the following cascade logic before starting the Node.js process:

1. If `DEFAULT_PROXY_API_KEY` is not set as an env var but exists in `/shared/proxy_api_key`, it is loaded from that file.
2. If `AXIOLOGIC_PROXY_API_KEY` is not explicitly set, it inherits the value of `DEFAULT_PROXY_API_KEY`.
3. If `ENCRYPTION_KEY` is not set but a key file exists at `/shared/soul-gateway/config/encryption.key`, it is loaded from that file.

> **Tip:** For provider-specific keys (Copilot, Kiro), the gateway sidecars handle their own authentication. The dummy values (`no-auth-needed`) are placeholders so achillesAgentLib recognizes these providers as available.

### Naming Conventions

**Model names** use the format `provider/model-name` without any gateway prefix. For example: `copilot/gpt-4o`, `anthropic/claude-sonnet-4`, `kiro/deepseek-3.2`.

**Tier names** use the `axl/` prefix to distinguish them from model names. For example: `axl/fast`, `axl/deep`, `axl/code`.

## Ploinky Manifest

The `manifest.json` file is the contract between Soul Gateway and the Ploinky orchestrator. It declares the container image, commands, environment, ports, dependencies, and installation hooks.

### Full Manifest Structure

```json
{
  "container": "node:20-slim",
  "agent": "bash /code/startup.sh",
  "cli": "bash /code/cli.sh",
  "about": "Soul Gateway -- LLM proxy with Soul Family policies, ...",
  "enable": ["postgres"],
  "profiles": {
    "default": {
      "env": { ... },
      "ports": ["0.0.0.0:8042:8042"],
      "install": "bash /code/install.sh"
    }
  }
}
```

### Top-Level Fields

| Field | Value | Description |
|-------|-------|-------------|
| `container` | `"node:20-slim"` | The OCI container image Ploinky pulls. Official Node.js 20 slim image based on Debian. |
| `agent` | `"bash /code/startup.sh"` | The entrypoint command Ploinky runs inside the container when starting the agent. |
| `cli` | `"bash /code/cli.sh"` | The command run when a user invokes `ploinky cli soul-gateway` for interactive CLI access. |
| `about` | *(string)* | Human-readable description shown in `ploinky list` and the dashboard. |
| `enable` | `["postgres"]` | Infrastructure services this agent depends on. Ploinky ensures PostgreSQL is running before starting Soul Gateway. Supported: `postgres`, `keycloak`. |

### Profiles

Profiles allow different configurations for different environments. Soul Gateway defines a single `default` profile that Ploinky uses unless overridden.

#### profiles.default.env

A map of environment variable declarations. Each entry has the structure:

```json
"VARIABLE_NAME": {
  "required": false,
  "default": "value",
  "description": "Human-readable description"
}
```

When `required` is `true`, Ploinky refuses to start the agent unless the variable is set (either in the manifest default, via `ploinky var`, or in `.ploinky/.secrets`). When `false`, the default value is used if the variable is not provided.

#### profiles.default.ports

Array of port mapping strings in the format `bind:host:container`:

```json
"ports": ["0.0.0.0:8042:8042"]
```

| Component | Value | Meaning |
|-----------|-------|---------|
| Bind address | `0.0.0.0` | Listen on all interfaces. In production behind Cloudflare tunnels, this can be narrowed to `127.0.0.1`. |
| Host port | `8042` | Port on the host machine. |
| Container port | `8042` | Port inside the container where the Node.js server listens. |

#### profiles.default.install

```json
"install": "bash /code/install.sh"
```

The install command runs once when Ploinky first creates the container (during `ploinky start` if no existing container is found). It creates persistent storage directories, copies application code, runs `npm install --production`, and generates the encryption key.

### How Ploinky Uses the Manifest

When you run `ploinky start soul-gateway`, the following sequence occurs:

1. **Dependency resolution:** Ploinky reads the `enable` array and ensures all listed services (e.g., postgres) are running.
2. **Container creation:** If no container exists, Ploinky pulls the `container` image (`node:20-slim`) and creates a new container with the declared env vars and port mappings.
3. **Variable injection:** Ploinky merges manifest defaults with `ploinky var` overrides and `.ploinky/.secrets`, then injects them as container environment variables.
4. **Volume mounts:** Ploinky mounts the repo directory at `/code` and the shared persistent storage at `/shared`.
5. **Install phase:** On first creation, Ploinky runs the `install` command inside the container.
6. **Agent start:** Ploinky executes the `agent` command (`bash /code/startup.sh`) as the container's main process.

> **Restart behavior:** When running `ploinky restart soul-gateway`, Ploinky first calls `syncCoreDependencies()` to rsync the latest achillesAgentLib from Ploinky's own `node_modules` into the agent's workspace. This ensures dependency updates propagate without a full container rebuild.

### Directory Layout Inside the Container

| Path | Purpose |
|------|---------|
| `/code/` | Read-only mount of the Soul Gateway source repo. Contains `app/`, `startup.sh`, `install.sh`, etc. |
| `/app/` | Working copy of the application. Code is copied here during install and startup to allow npm to write `node_modules`. |
| `/shared/soul-gateway/` | Persistent storage that survives container recreation. |
| `/shared/soul-gateway/config/encryption.key` | Auto-generated AES-256 encryption key (64 hex characters). |
| `/shared/proxy_api_key` | Optional file containing the default proxy API key, loaded by `startup.sh`. |

## Deployment

### install.sh

The install script runs once when Ploinky first creates the container. It sets up the filesystem and dependencies needed before the application can start.

```bash
#!/bin/bash
set -e
APP_DIR="/app"
CODE_DIR="/code"
SHARED_DIR="/shared/soul-gateway"
```

**What it does:**

1. **Creates persistent directories:** `/shared/soul-gateway/config/` and `/app/`. The `/shared` directory is a Ploinky-managed volume that survives container recreation.
2. **Copies application code:** Copies from `/code/app/` (the repo mount) to `/app/`. Falls back to the workspace repos path (`$WORKSPACE_PATH/.ploinky/repos/proxies/soul-gateway/app/`) if `/code` is not available.
3. **Installs npm dependencies:** Runs `npm install --production` in `/app/` if a `package.json` exists.
4. **Generates encryption key:** If no key file exists at `/shared/soul-gateway/config/encryption.key`, generates a cryptographically random 32-byte hex key using Node.js `crypto.randomBytes(32)`. This key is used for AES-256 encryption of API keys in the database.

> **Important:** The encryption key is generated once and persisted to shared storage. Losing this key means all encrypted API keys in the database become unrecoverable. Back up `/shared/soul-gateway/config/encryption.key`.

### startup.sh

The startup script runs every time the container starts. It ensures the latest code is deployed, secrets are loaded, and the Node.js process begins.

**Startup sequence:**

1. **Ensure directories:** Creates `/shared/soul-gateway/config/` if missing.
2. **Copy/update application:** Copies from `/code/app/` to `/app/`. This means every restart picks up the latest code from the repo mount. Falls back to the workspace repos path if `/code` is not available.
3. **Install dependencies:** If `node_modules` is missing in `/app/`, runs `npm install --production`.
4. **Load encryption key:** If `ENCRYPTION_KEY` env var is empty, reads from `/shared/soul-gateway/config/encryption.key`.
5. **Load proxy API key:** If `/shared/proxy_api_key` exists, reads it into `DEFAULT_PROXY_API_KEY`.
6. **Cascade API keys:** Sets `AXIOLOGIC_PROXY_API_KEY` from `DEFAULT_PROXY_API_KEY` if not already set.
7. **Start the application:** Runs `exec node src/index.mjs` in `/app/`. The `exec` replaces the shell process so Node.js receives signals directly.

### Application Boot Order

Once `node src/index.mjs` starts, the application initializes in a strict sequence defined in `main()`:

| Step | Function | Description |
|------|----------|-------------|
| 1 | `initDb()` | Creates the `soul_gateway` PostgreSQL schema, runs DDL migrations to create or update all tables, and seeds initial data (default tiers like `axl/fast`, `axl/deep`, `axl/code`, and provider configs). |
| 2 | `scanMiddlewares()` | Scans the `src/middlewares/` directory, imports all `.mjs` files, validates the middleware interface, and upserts them into the `middlewares` database table. |
| 3 | `registerAdapter()` (x6) | Registers OAuth/auth adapters for Copilot, Kiro, Codex, Gemini, Anthropic, and Search providers. Each adapter defines how to authenticate, refresh tokens, and provision models. |
| 4 | `startRefreshLoop()` | Starts a background timer that periodically refreshes OAuth tokens for all registered providers with stored credentials. |
| 5 | `reconcileProviders()` | Auto-provisions provider and model config rows in the database for any adapter that has stored credentials but no matching `provider_configs` row. Ensures newly authenticated providers are immediately available. |
| 6 | `createAppServer()` | Creates the HTTP server with all route handlers (chat completions, models, keys, providers, tiers, middlewares, logs, metrics, dashboard, health, WebSocket). |
| 7 | `startServer()` | Binds the HTTP server to the configured port and begins accepting requests. |

> **Fatal errors:** If any step in the boot sequence throws an exception, the process logs a critical error and exits with code 1. Ploinky will not automatically restart the container -- check logs with `ploinky logs soul-gateway`.

### deploy.sh

The `deploy.sh` script is called by the GitHub Actions workflow `deploy-soul-gateway.yml` on the production server. It handles the full deployment lifecycle.

**Deployment steps:**

1. **Clone or pull repos:** Fetches the latest code from `PloinkyRepos/proxies` (Soul Gateway source) and `PloinkyRepos/Basic` (infrastructure manifests like postgres) into `~/code/`.
2. **Create workspace:** Ensures the `~/soulGateway` workspace directory exists.
3. **Register repos:** Creates symlinks from `.ploinky/repos/` to the cloned source directories. Writes `enabled_repos.json` with `["basic","proxies"]` if not present.
4. **Configure variables:** Sets environment variables via `ploinky var` -- `UPSTREAM_URL`, `PGHOST`, `PGPORT`, `PGUSER`, `PGPASSWORD`, `PGDATABASE`, and `DEFAULT_PROXY_API_KEY`. Sensitive values (`PGPASSWORD`, `DEFAULT_PROXY_API_KEY`) come from GitHub Actions secrets.
5. **Stop existing:** Runs `ploinky stop soul-gateway` and `ploinky clean soul-gateway` to remove the existing container.
6. **Start fresh:** Runs `ploinky start soul-gateway`, which triggers install and then startup.
7. **Health check:** Polls `GET /health` every 2 seconds for up to 60 seconds (30 attempts). Exits with success on first healthy response, or failure with the last 20 container log lines.

**Triggering a deploy:**

Production deployments should always be triggered via GitHub Actions, never via direct SSH:

```bash
gh workflow run deploy-soul-gateway.yml -R PloinkyRepos/proxies
```

The workflow also auto-triggers on pushes to `soul-gateway/*` paths in the proxies repo.

### Health Check

**`GET /health`**

Returns a JSON response indicating the gateway's health status. Used by deploy scripts, monitoring systems, and Cloudflare tunnel health probes.

```bash
curl http://localhost:8042/health
```

A successful response indicates the HTTP server is running and accepting connections. The deploy script considers any 2xx response as healthy.

### Graceful Shutdown

Soul Gateway handles `SIGTERM` and `SIGINT` signals for clean shutdown. The `exec` in `startup.sh` ensures signals reach the Node.js process directly (not the parent shell).

**Shutdown sequence:**

1. Signal received (`SIGTERM` from Ploinky stop, or `SIGINT` from Ctrl+C).
2. **Stop refresh loop:** Cancels the background OAuth token refresh timer.
3. **Close server:** Stops accepting new connections and waits for in-flight requests to complete.
4. **Log and exit:** Logs "Server closed" and exits with code 0.
5. **Force exit:** If the server does not close within 10 seconds, the process exits with code 1 to prevent hanging.

> **In-flight requests:** The 10-second timeout allows streaming responses to finish. Long-running requests (e.g., large code generation) may be interrupted if they exceed this window.
